# Журнал архитектурных решений

## 2026-06-07

### Решение

Flet в текущей архитектуре используется только как нативная WebView-оболочка и отвечает за системные состояния запуска: ввод адреса приложения в dev-режиме, загрузку, отсутствие сети, ошибку сервера, повторную попытку и портретную ориентацию телефона.

### Причина

Основной интерфейс перенесён на React/Vite, поэтому наращивать продуктовый UI во Flet больше нельзя. Но WebView-оболочка должна вести себя как приложение: пользователь не должен видеть технические ошибки, `vite`, `React`, `WebView`, traceback или инструкции разработчика при отсутствии сети.

### Последствия

`src/mobile_webview.py` содержит только минимальный shell-слой. Проверка доступности сервиса и сообщения об ошибках отделены от React-интерфейса. Портретная ориентация включается нативно через Flet `set_allowed_device_orientations` только для телефонного iOS/Android viewport; финальное подтверждение поведения остаётся ручной проверкой на реальном устройстве.

## 2026-06-04

### Решение

Для публичного контента React/Vite использовать стратегию `API-first` с mock fallback.

### Причина

React-версия уже читает публичные проблемы, сценарии, документы, учреждения и закон-апдейты из FastAPI, но часть backend summary DTO пока беднее старых mock-данных. Простое склеивание списков создаёт дубли, а полная замена ломает preview-карточки и локальные ситуации, завязанные на mock-сценарии.

### Последствия

Store React-версии нормализует публичные элементы по названию и категории: backend-элемент получает приоритет, одноимённая mock-карточка убирается или используется как временный источник богатых preview-массивов. Mock остаётся fallback при недоступном API. Следующий backend-шаг — расширить summary DTO сценариев, чтобы каталог React больше не зависел от mock preview.

## 2026-06-03

### Решение

Начать постепенный перенос пользовательского интерфейса на Vite/React/TypeScript, используя папку `reactvitemaket/` как frontend-кандидат, при этом сохранить Flet UI как рабочий baseline до достижения функционального parity.

### Причина

Для проекта нужен более гибкий, адаптивный и визуально сильный интерфейс по Figma. Создание сложного UX во Flet стало рискованным и затратным: часть экранов сложно полировать, появляются платформенные баги, а будущая web/mobile адаптация проще контролируется в React.

### Последствия

FastAPI backend и накопленная бизнес-логика сохраняются. Новая миграция должна идти поэтапно: сначала стабилизировать React/Vite-макет, затем подключать API и переносить модули один за другим. Flet UI нельзя удалять или ломать до отдельного решения. Зафиксированы документы `docs/REACT_MIGRATION_AUDIT.md` и `docs/REACT_MIGRATION_PLAN.md`.

## 2026-05-23

### Решение

Закон-апдейты на текущем этапе редактировать локально через Flet-админку и хранить в `data/app_state.json`, не подключая production-уведомления и RBAC.

### Причина

ТЗ требует, чтобы редактор мог создавать и публиковать изменения законодательства. Для постепенного закрытия ТЗ достаточно сначала показать рабочую локальную модель: черновик, публикация, источник, приоритет и аудит действия редактора.

### Последствия

Лента закон-апдейтов теперь читает опубликованные записи из локального состояния, а админ-панель позволяет создать, опубликовать/вернуть в черновик и удалить запись. Редактирование всех полей, связь со сценариями, релевантность по профилю и production-уведомления остаются следующими задачами.

## 2026-05-23

### Решение

Роли `Гражданин`, `Редактор контента` и `Админ площадки` сначала реализовать как локальный демонстрационный слой в Flet MVP, без настоящего backend RBAC.

### Причина

ТЗ требует разделить роли и показать, что редактор контента и администратор площадки имеют разные зоны ответственности. Для текущего этапа нужно продолжать закрывать внутреннюю логику приложения, не переписывая авторизацию и backend раньше времени.

### Последствия

В локальном состоянии появились `admin_roles`, `admin_users`, `admin_audit_logs` и `current_admin_role`. Админ-панель показывает права и доступные разделы выбранной роли. Production RBAC, управление пользователями, полноценный аудит и серверные проверки доступа остаются отдельным backend-этапом.

## 2026-05-20

### Решение

Справочник госучреждений для ближайшего MVP реализовать локально в `src/data/mock_data.py`, а подбор вынести в сервис `src/services/institutions.py`.

### Причина

ТЗ требует показывать пользователю, куда обращаться, с учётом его населённого пункта. Для демонстрационного MVP достаточно локального справочника и простого подбора по типу учреждения, региону и городу без GPS, карт и production-адресной базы.

### Последствия

Задачи сценариев получили `institution_types`, а детальная ситуация показывает блок «Куда обращаться» на основе профиля пользователя. Админский CRUD учреждений, точные официальные адреса и геопоиск остаются следующими этапами.

## 2026-05-20

### Решение

Зависимости шагов сценариев сначала реализовать в локальном MVP через поле `depends_on` у задач шаблона и локальные ID задач пользовательской ситуации.

### Причина

ТЗ требует вести пользователя по правильному маршруту, где часть действий нельзя выполнить раньше предыдущих шагов. Для быстрой демонстрации этой логики не нужен новый backend-этап: достаточно сохранять зависимости в моковых шаблонах и локальном состоянии.

### Последствия

При создании ситуации из шаблона зависимости переводятся в локальные ID задач. Детальная ситуация показывает заблокированные шаги и причину блокировки, а отметка выполнения запрещена до выполнения зависимостей. Управление зависимостями через админ-панель остаётся следующим этапом.

## 2026-05-16 (финальный функциональный проход)

### Решение

Авторизацию и пользовательские действия в мобильном UI реализовать как локальный MVP в текущей сессии, без JWT, паролей в базе и RBAC.

### Причина

Пользователю нужен функциональный дипломный прототип уже сейчас: вход, регистрация, добавление документов/ситуаций/задач, сохранение профиля и уведомления. Полноценная безопасная авторизация требует отдельного backend-контракта и не должна смешиваться с текущим UI-инкрементом.

### Последствия

Появились рабочие `LoginPage` и `RegisterPage`, а основные кнопки стали интерактивными. На следующем проходе локальные данные были вынесены в демо-хранилище `data/app_state.json`; перенос на backend/JWT остается отдельным этапом.

## 2026-05-16 (локальное сохранение MVP)

### Решение

Сохранять локальное состояние мобильного MVP в `data/app_state.json`, не превращая это в полноценную backend-авторизацию.

### Причина

Пользовательские действия должны переживать перезапуск приложения во время демонстрации дипломного MVP: добавленные документы, ситуации, задачи, профиль и прочитанные уведомления не должны исчезать сразу после закрытия окна.

### Последствия

Приложение стало ближе к рабочему прототипу, но файл `data/app_state.json` остается локальным демо-хранилищем. Для production-версии по-прежнему нужен backend, JWT/RBAC и серверное хранение пользовательских данных.

## 2026-05-18

### Решение

Перед каждой записью `data/app_state.json` создавать локальный backup `data/app_state.backup.json`.

### Причина

Локальное JSON-хранилище может быть повреждено при ручном редактировании или аварийном завершении записи. Для дипломного MVP лучше восстановиться из последнего рабочего состояния, чем сбросить пользовательские данные.

### Последствия

`LocalStore` сначала читает основной JSON, при ошибке пробует backup и восстанавливает основной файл. Это не заменяет полноценную базу данных, но повышает устойчивость локального режима.
