# Аудит React/Vite-макета для миграции UI

Дата: 2026-06-03.

Документ фиксирует текущее состояние папки `reactvitemaket/` и показывает, что нужно подготовить перед переносом логики из Flet/Python UI в React/Vite frontend.

## 1. Назначение аудита

`reactvitemaket/` рассматривается как новый frontend-кандидат для проекта «Белпомощник».

Важно:

- это пока не замена рабочему приложению;
- рабочая версия проекта остаётся доступна через Flet/Python;
- backend FastAPI + SQLite сохраняется;
- перенос должен идти поэтапно, без массового переписывания всей логики за один раз.

## 2. Что есть в React/Vite-макете

Стек макета:

- Vite;
- React;
- TypeScript;
- React Router;
- Tailwind CSS;
- Radix/shadcn-подобные UI-компоненты;
- `lucide-react`;
- локальный store через React state.

После первичного аудита `package.json` был сокращён до реально используемых зависимостей текущего reachable bundle: `react`, `react-dom`, `react-router`, `motion`, `lucide-react`, `tw-animate-css`, Vite и Tailwind. Неиспользуемые Radix/MUI/Recharts/DnD-пакеты не удалены из исходной папки компонентов, но больше не тянутся установщиком, пока эти компоненты не подключены к приложению.

Основные файлы:

| Файл | Назначение |
|---|---|
| `reactvitemaket/src/main.tsx` | точка входа React-приложения |
| `reactvitemaket/src/app/App.tsx` | общий shell, desktop/mobile layout, часть экранов |
| `reactvitemaket/src/app/routes.tsx` | карта React Router маршрутов |
| `reactvitemaket/src/app/pages.tsx` | основные страницы |
| `reactvitemaket/src/app/components/extra-screens.tsx` | детальные страницы, настройки, обучение, поиск, модалки |
| `reactvitemaket/src/app/data/types.ts` | TypeScript-типы доменной модели |
| `reactvitemaket/src/app/data/mock.ts` | моковые сценарии, документы, уведомления, закон-апдейты |
| `reactvitemaket/src/app/data/store.tsx` | локальное состояние React-макета |

## 3. Что можно сохранить из текущего проекта

При переходе на React/Vite сохраняем:

- backend FastAPI в `src/backend`;
- API-модели сценариев, документов, пользователей, ситуаций и трекеров;
- миграции и seed-данные;
- тестовые аккаунты и роли;
- документы ТЗ, roadmap и отчёты;
- бизнес-логику, которая уже вынесена в сервисы;
- структуру жизненных сценариев: сценарии, этапы, задачи, документы, учреждения, источники, зависимости;
- текущую логику прав доступа как основу для frontend RBAC;
- накопленную UX-логику: гостевой режим, быстрые роли, защищённые действия, документы, уведомления, профиль, ЖКХ/налоги.

Flet UI временно остаётся как рабочий baseline до достижения parity в React.

## 4. Сильные стороны React/Vite-макета

- Есть отдельные desktop и mobile раскладки.
- Есть современный визуальный стиль.
- Есть базовые сущности проекта: сценарии, документы, ситуации, профиль, уведомления, закон-апдейты.
- Есть route-структура, близкая к текущему приложению.
- Есть роли `guest`, `citizen`, `editor`, `admin`.
- Есть локальная логика создания ситуации из сценария и расчёта прогресса.
- Есть заготовка защищённых действий для гостя.

## 5. Главные проблемы перед переносом логики

React-макет нельзя сразу считать рабочим приложением. Перед подключением backend нужно стабилизировать модель данных и маршруты.

### 5.1. Несовпадения типов и данных

| Место | Проблема | Что сделать |
|---|---|---|
| `Category` | тип содержит `name`, часть UI читает `title` | привести UI к `name` или расширить тип |
| `UserDocument` | тип содержит `title`, часть UI читает `name` | унифицировать на `title` |
| `UserDocument.status` | тип: `active/expiring/expired`, часть UI проверяет `soon` | заменить `soon` на `expiring` |
| `AppNotification` | тип содержит `kind`, часть UI читает `type` | унифицировать на `kind` |
| `LegalUpdate.priority` | тип boolean, часть UI сравнивает со строкой `"high"` | заменить на `priority: low/medium/high` или привести UI к boolean |
| `LawDetailPage` | читает поля `content`, `forWhom`, которых нет в типе | использовать `whatChanged`, `whoAffected`, `whatToDo` |

### 5.2. Несовпадения маршрутов

| Требуемый смысл | Сейчас в React | Что сделать |
|---|---|---|
| Список жизненных сценариев | нет отдельного `/scenarios` index | добавить `/scenarios` |
| Деталь сценария | `/scenarios/:id` | оставить |
| Админ-панель | компонент есть частично, маршрута нет | добавить `/admin` с проверкой роли |
| Редакторский раздел | нет отдельного маршрута | добавить позже `/editor` или вкладки внутри admin |
| Каталог проблем | `/catalog` и `/problems` частично дублируются | выбрать основную схему и оставить алиасы |

### 5.3. Архитектурные проблемы макета

- `App.tsx` слишком большой и смешивает shell, mobile UI, desktop UI и часть экранов.
- В `App.tsx` встречается повторный импорт React внутри файла.
- Моковые данные и UI местами связаны напрямую.
- Темизация частично идёт через CSS-переменные, но многие цвета захардкожены в компонентах.
- Админ-панель из `desktop.tsx` пока выглядит как отдельная визуальная заготовка и не встроена в маршруты.
- Login/register пока локальные и не подключены к backend auth.
- Поиск и подсказки работают локально, без API.
- Нет полноценного API-клиента и contract layer.

## 6. Что нельзя потерять при миграции

При переносе на React нельзя потерять:

- гостевой режим;
- быстрый вход по ролям;
- корректный выход и очистку сессии;
- защищённые действия для гостя;
- создание ситуации из шаблона;
- зависимости задач;
- документы внутри задач;
- личные документы и приватность;
- закон-апдейты с источниками;
- подбор учреждений по профилю;
- ЖКХ/налоги;
- локализацию RU/EN/BE, когда она будет внедряться;
- админские и редакторские права.

## 7. Риски миграции

- Можно быстро получить красивый UI, но потерять уже сделанную бизнес-логику.
- Есть риск дублировать моковые данные вместо подключения к API.
- TypeScript-ошибки в макете могут заблокировать сборку.
- Если сразу переносить все экраны, сложно понять, где сломался сценарий.
- Если оставить Flet и React параллельно без правил, документы и логика начнут расходиться.

## 8. Рекомендация

Идти по безопасной схеме:

1. Сначала стабилизировать React/Vite-макет: типы, маршруты, layout, темы.
2. Добавить API-клиент и адаптеры данных.
3. Подключать backend по одному модулю: публичные сценарии, авторизация, профиль, ситуации, документы, уведомления, админка.
4. Flet UI держать как рабочий fallback до тех пор, пока React не повторит основной пользовательский путь.
5. После parity выбрать упаковку: PWA, Capacitor, Tauri/Electron или Flet WebView-shell.

## 9. Следующий технический шаг

Первый кодовый шаг миграции после успешной сборки:

- исправить несовпадения типов;
- добавить маршруты `/scenarios`, `/admin`;
- создать `src/app/services/api.ts`;
- создать слой адаптеров `src/app/data/adapters.ts`;
- не подключать сразу все API, а оставить mock fallback.
