## Текущий курс UI (исторический раздел — Flet-этап)

- Рабочий baseline проекта остаётся на Flet/Python.
- Папка `reactvitemaket/` принята как frontend-кандидат для постепенного переноса UI на Vite/React/TypeScript.
- Backend FastAPI + SQLite MVP сохраняется и должен стать источником данных для React frontend через API.
- Flet UI не удаляется и не ломается до достижения функционального parity в React.
- Созданы документы:
  - `docs/REACT_MIGRATION_AUDIT.md`;
  - `docs/REACT_MIGRATION_PLAN.md`.
- `reactvitemaket/package.json` сокращён до реально используемых зависимостей, чтобы первый frontend запуск не тянул неиспользуемый Radix/MUI/Charts-набор.
- Зависимости `reactvitemaket/` установлены через pnpm, создан `pnpm-lock.yaml`.
- `pnpm build` для React/Vite-макета проходит без ошибок.
- React/Vite dev-сервер проверен на `http://127.0.0.1:8560/`: главная страница открывается без runtime traceback.
- Добавлены маршруты React `/scenarios` и `/admin`; `/admin` закрыт role guard для не-администратора.
- Создана начальная frontend-основа для API-интеграции: `reactvitemaket/src/app/services/api.ts` и `reactvitemaket/src/app/data/adapters.ts`.
- React/Vite store подключен к публичным backend endpoints `/api/problems` и `/api/scenarios` с mock fallback.
- Детальная страница React-сценария умеет догружать полный сценарий через `/api/scenarios/{slug}`.
- React/Vite store подключен к публичным endpoints `/api/documents`, `/api/authorities` и `/api/law-updates`.
- Глобальный поиск React-версии теперь видит справочные документы и учреждения из backend API.
- Для публичного React-контента выбрана стратегия `API-first`: backend-данные заменяют одноимённые mock-карточки, mock остаётся fallback при недоступном API.
- Backend summary DTO сценариев расширен полями `stage_count` и `task_count`; React-каталог показывает число задач из API, а mock-массивы остаются только fallback для подробного preview.
- В React/Vite добавлена локальная основа аккаунтов: гость, гражданин, редактор, администратор, быстрый вход, регистрация нового локального пользователя, выход в гостевой режим и protected actions для личных действий.
- В React/Vite добавлено локальное per-user хранение личных данных: ситуации, задачи, документы, избранное, уведомления, профиль и настройки сохраняются отдельно для каждого аккаунта.
- Гостевой режим в React/Vite получает пустую личную область и не может создавать ситуации, отмечать задачи или добавлять документы.
- В React/Vite добавлен мягкий backend auth bridge: тестовые и локально зарегистрированные аккаунты пытаются получить JWT через `/api/auth/login` или `/api/auth/register`, при недоступном backend сохраняется локальный режим.
- Личные документы React/Vite подключены к backend endpoint `/api/user/documents`: список документов догружается из API, создание/редактирование/удаление backend-документов синхронизируются с API, локальный fallback остаётся рабочим.
- Пользовательские ситуации и задачи React/Vite подключены к backend endpoints `/api/user/situations` и `/api/user/tasks`: после JWT-входа ситуации догружаются из API, создание ситуации из сценария создаёт backend-ситуацию с задачами, отметка задачи синхронизируется через API, локальный fallback остаётся рабочим.
- В backend добавлен публичный endpoint `GET /api/scenarios`, CORS для React dev-сервера и категория в DTO полного сценария.
- Локальная backend-база обновлена миграциями `0003`-`0006` и заполнена MVP seed-данными сценария «Рождение ребёнка».
- Следующий маленький шаг: подключать профиль, настройки и уведомления React-версии к backend API, сохраняя текущий локальный fallback.

## Известные проблемы

- Глобальная команда `flet` может быть недоступна без активации `.venv`.
- Команды `python` и `pip` в системе могут отсутствовать; использовать `python3`, `pip3` или команды из `.venv`.
- Поиск и фильтрация реализованы только на моковых данных.
- Авторизация реализована как локальный MVP без реальной безопасности, JWT и RBAC.
- Для работы админ-функций Flet-экрана backend API должен быть запущен отдельно (`uvicorn backend.app:app ...`).
- Обучающий модуль реализован как демо/MVP на локальном состоянии без backend.
- Backend добавлен как MVP-основа, Flet-клиент переведен на API только частично; React/Vite frontend-кандидат уже читает публичные проблемы и сценарии через API с fallback.
- В конструкторе сценариев пока нет drag-and-drop сортировки; порядок задается числом `order_index`.
- В админке пока нет удаления сущностей и массовых операций.
- Локальные пользовательские данные сохраняются в `data/app_state.json`, но это не заменяет backend-хранилище и реальную безопасность.
- Личные документы в демо-версии хранятся локально без шифрования; это допустимо для MVP, но не для production.
- На некоторых платформах Flet может не отдать локальный путь выбранного файла; в этом случае поле «Файл / скан» можно заполнить вручную для демо.
- Справочник учреждений демонстрационный: адреса и телефоны требуют финальной проверки по официальным источникам перед production.
- Шаблоны сценариев пока моковые; позже их нужно получать из public API сценариев.
- `flet run src/main.py` запускает долгоживущий GUI-процесс; для остановки использовать `Ctrl+C`.
- При запуске `flet run -w ...` важно использовать свободный порт; если порт занят, нужно выбрать другой.
- Android-адаптация и планшетные раскладки не выполнялись в текущем iPhone-этапе.
- На реальном iPhone нужно дополнительно проверить поведение клавиатуры в диалогах редактирования документов, ситуаций и задач.

## Команды запуска

С активацией окружения:

```bash
source .venv/bin/activate
flet run src/main.py
```

Без активации окружения:

```bash
.venv/bin/flet run src/main.py
```

Запуск в браузере для проверки внешнего вида:

```bash
.venv/bin/flet run -w src/main.py --host 127.0.0.1 -p 8550
```

Запуск на подключенное iOS-устройство, если Flet CLI и устройство уже настроены:

```bash
.venv/bin/flet run --ios src/main.py
```

Инициализация БД сценариев:

```bash
PYTHONPATH=src python -m backend.scripts.init_db
PYTHONPATH=src python -m backend.scripts.seed_db
```

Запуск API:

```bash
PYTHONPATH=src uvicorn backend.app:app --host 127.0.0.1 --port 8060
```

### Сознательно отложено в «Перспективы» (production TODO)

> Эти пункты **не блокируют защиту**. Они описаны здесь, чтобы на вопрос «а почему не…» был честный ответ.

- **Refresh-токены в БД** — хранятся как есть (в `refresh_tokens.token` поле). Для production — SHA-256 + индекс; refresh-rotation уже работает.
- **Login rate-limit** — in-memory (`collections.deque`). Для production — Redis (`slowapi` или свой backend).
- **CORS** — `allow_origin_regex` открывает любой LAN IP (намеренно для iPhone WebView). В production — strict allow-list.
- **Document scan ACL** — `GET /api/user/documents/{id}/scan` (новый) требует auth, но `app.mount("/uploads/documents", …)` оставлен для обратной совместимости. TODO: переключить UI на новый endpoint, удалить mount.
- **Article body_html** — регулярная санитизация от `<script>`/on*-attrs. Полноценный sanitize — `bleach`/`DOMPurify`.
- **Settings allow-list** — покрывает известные ключи; новые — добавлять явно.
- **Frontend: пароли в LS** — теперь санитизируются перед записью. Для production — отдельный sessionStorage + ввод пароля в UI.
- **Audit log** — пишет create/delete, **не пишет update** для проблем/сценариев/документов/закон-апдейтов. Дополнить.
- **Push-уведомления** — backend endpoint есть, фронт использует Web Notifications API. Для нативной APK/iOS-сборки — FCM/APNs.
- **AI-ассистент** — Groq API. Без ключа — canned-ответы. Не критично для демо.
- **Email-доставка** — SMTP-очередь. Без SMTP_* env-ов — email'ы зависают в `email_notifications` со статусом `pending`.
- **Мобильный адаптив** — есть, но часть legacy-компонентов в `App.tsx` (`MobileHome`/`DesktopHome` и т.п.) не используется роутингом — это каркас, оставлен для рефакторинга.
