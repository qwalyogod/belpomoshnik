# K7. Ограничения MVP

## 7.1 Что реализовано в MVP

«Белпомощник» — дипломный MVP, демонстрирующий все ключевые концепции системы в работающем виде. Следующие функции реализованы полностью:

- Каталог из 14 проблем и 9 жизненных сценариев с этапами, шагами, документами.
- Пошаговое прохождение сценария с зависимостями шагов.
- Персонализация через профиль пользователя.
- Трекеры ЖКХ и налоговых обязательств.
- Уведомления о сроках документов, ЖКХ, налогов.
- Закон-апдейты с привязкой к категории профиля.
- Полная административная панель (CRUD всего контента).
- Backend REST API с JWT-авторизацией и RBAC.
- Audit trail (журнал действий).
- Демо-предпросмотр email-уведомлений.

---

## 7.2 Ограничения и причины

### OG-01. Email и push — только MVP-демонстрация

**Ограничение:** Email-уведомления не отправляются в реальном времени. Функция `send_pending_emails()` реализована, но SMTP-сервер не настроен. Push-уведомления (Web Push / FCM) не реализованы.

**Причина:** В дипломном MVP недостаточно инфраструктуры (нет production-сервера, домена, SMTP-аккаунта). Реализация ограничена демо-предпросмотром письма в UI и архитектурным описанием (`docs/PUSH_NOTIFICATIONS.md`).

**Путь в production:** Добавить переменные среды `SMTP_HOST`, `SMTP_USER`, `SMTP_PASSWORD`, запустить планировщик `send_pending_emails(db)` через APScheduler или cron.

---

### OG-02. SQLite вместо PostgreSQL

**Ограничение:** Backend использует SQLite (файл `belpomoshnik.db`). При одновременной нагрузке SQLite ограничен по параллельным записям.

**Причина:** PostgreSQL требует отдельной установки и настройки, что усложняет воспроизводимость дипломного проекта.

**Путь в production:** Изменить строку подключения `DATABASE_URL` на `postgresql+asyncpg://...`. SQL-миграции написаны в стандартном SQL и совместимы с PostgreSQL.

---

### OG-03. Нет интеграции с государственными API

**Ограничение:** Приложение не подключено к ЕГИАС, portal.gov.by или другим реальным государственным реестрам.

**Причина:** Государственные API Республики Беларусь не имеют открытой документации для разработчиков и требуют официального согласования.

**Путь в production:** После официального получения доступа — добавить API-клиент в `src/backend/integrations/`.

---

### OG-04. Однопользовательская архитектура хранения файлов

**Ограничение:** Загруженные файлы документов хранятся локально в `data/private_uploads/` без разграничения по пользователям и без CDN.

**Причина:** MVP не требует production-хранилища файлов.

**Путь в production:** Использовать MinIO или S3-совместимое хранилище с presigned URL. Маршрут `/api/me/documents/{id}/file` уже реализован как заготовка.

---

### OG-05. Нет мультиязычности

**Ограничение:** Интерфейс только на русском языке. Беларусский язык не поддерживается.

**Причина:** Временные ограничения дипломного проекта. Добавление i18n потребует рефакторинга всех строк в страницах.

---

### OG-06. Нет offline-режима для мобильного приложения

**Ограничение:** При недоступном backend клиент использует mock-данные из `mock_data.py`, но не синхронизирует изменения при восстановлении связи.

**Причина:** Реализация offline-first потребует conflict resolution и очереди синхронизации — за рамками дипломного MVP.

---

### OG-07. Нет реального OAuth / ЕСИА

**Ограничение:** Авторизация через логин/пароль. Интеграции с Белорусской системой идентификации (ЕСИА или аналог) нет.

**Причина:** ЕСИА Беларуси не предоставляет публичного OAuth-протокола для внешних приложений.

---

### OG-08. Данные сценариев — справочные

**Ограничение:** Тексты шагов и документов составлены на основе публично доступных источников и проверены на май 2026 года. Законодательство меняется; для актуальных данных необходима регулярная сверка.

**Путь:** Встроенная система закон-апдейтов позволяет редактору публиковать изменения без выпуска новой версии приложения.

---

### OG-09. Нет реального расписания уведомлений

**Ограничение:** Уведомления о сроках документов и ЖКХ рассчитываются локально при старте приложения (`sync_task_notifications()` в `main.py`), а не через фоновый сервис.

**Путь в production:** Вынести в APScheduler или отдельный worker-процесс, запускаемый по cron.

---

### OG-10. Desktop-UI, не нативное мобильное приложение

**Ограничение:** Приложение запускается через Flet (`flet run` или web-версия). Нативные iOS/Android пакеты не собраны.

**Путь:** Flet поддерживает сборку под Android (`flet build apk`) и iOS (`flet build ipa`) — требуется дополнительная настройка Flutter SDK.
