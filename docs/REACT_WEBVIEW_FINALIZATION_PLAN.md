# План доводки React/WebView-версии по ТЗ

Дата фиксации: 2026-06-07.

Документ фиксирует новый актуальный курс проекта после переноса UI на React/Vite и удаления нативного Flet-интерфейса. Его нужно читать перед крупными задачами по React/WebView вместе с `AGENTS.md`, `docs/PROJECT_STATUS.md`, `docs/TASKS.md`, `docs/REACT_MIGRATION_PLAN.md` и `docs/REACT_MIGRATION_AUDIT.md`.

## 1. Текущая архитектура

Актуальная структура продукта:

```text
reactvitemaket/
  React + Vite + TypeScript
  основной пользовательский интерфейс для web, desktop browser и WebView

src/backend/
  FastAPI + SQLAlchemy + SQLite MVP
  API, роли, пользовательские данные, публикации, трекеры, админ-эндпоинты

src/mobile_webview.py
  единственный оставшийся Flet-файл
  тонкая WebView-оболочка для показа React-сайта на мобильном устройстве

landing/
  отдельный лендинговый React-макет, визуальный референс для /welcome

docs/
  ТЗ, roadmap, отчеты, планы переноса и дипломная документация
```

Нативный Flet UI больше не является рабочим интерфейсом. Flet допускается только как оболочка, которая открывает React-сайт и решает нативные вопросы: загрузка, ошибка сети, недоступность сервера, ориентация телефона, будущая приватность приложения и биометрия.

## 2. Главный принцип дальнейшей работы

Сначала доводим продукт до надежного web/WebView-приложения, потом углубляем контент.

Приоритеты:

1. Стабильный запуск и понятные ошибки в WebView.
2. Исправление сломанных маршрутов и экранов, которые мешают демонстрации.
3. Единая навигация и понятное разделение проблем, сценариев и ситуаций.
4. Реальные пользовательские функции: профиль, адреса, документы, уведомления, источники.
5. Честная админка: показывать, что реально считается на backend, а что еще требует доработки.
6. Контент и официальные источники добавлять только после отдельной проверки.

Не делаем:

- не возвращаем нативный Flet UI;
- не маскируем WebView проблемами React-интерфейса;
- не добавляем юридически рискованный контент без официальных источников;
- не называем пользовательский UI демо-режимом;
- не делаем fake Face ID / Touch ID внутри сайта;
- не переписываем backend-стек без отдельного решения.

## 3. Быстрый аудит текущего состояния

### Что уже есть

- React/Vite-приложение с маршрутами `/`, `/welcome`, `/catalog`, `/problems`, `/scenarios`, `/situations`, `/documents`, `/finance`, `/news`, `/sources`, `/notifications`, `/profile`, `/settings`, `/admin`, `/editor`, `/login`, `/register`.
- FastAPI backend с публичными API, auth, пользовательскими ситуациями, документами, уведомлениями, трекерами ЖКХ/налогов, публикациями и админ-эндпоинтами.
- Гостевой режим, тестовые роли, быстрый вход, локальная регистрация, local fallback и backend auth bridge.
- Пользовательские ситуации, задачи, документы, профиль, настройки и уведомления частично синхронизируются с backend.
- Публикации и просмотры статей имеют backend API, но часть аналитики админки остается представительской.
- `landing/` уже содержит отдельный визуальный референс, на который нужно ориентировать `/welcome`.
- `src/mobile_webview.py` уже открывает React URL и имеет dev-поле ввода адреса.

### Что найдено как проблемное

- В `SettingsPage` виден лишний закрывающий `</div>` в JSX. Это нужно проверить первым: возможная причина нестабильной первой загрузки настроек.
- Карточки закон-апдейтов в `NewsPage` ведут на `/legal`, который редиректит на `/news`, вместо детальной страницы `/law-detail/:id`.
- `/sources` пока отдельный маршрут и отдельный пункт, но по новой логике источники должны жить внутри раздела «Новости».
- В навигации нужно поменять «Главная» и «Каталог» местами во всех меню.
- `/welcome` визуально близок, но еще не повторяет богатство `landing/` и на desktop есть ощущение обрезанного фона.
- Login/Register пока mobile-first; desktop/tablet требуют split-layout: слева смысл/преимущества, справа форма.
- Профиль слишком пустой: нет загрузки аватарки, пользовательских заметок, дополнительных адресов и предпочтений источников.
- Подбор учреждений сейчас идет по одному профилю/адресу; нужно поддержать до 5 адресов и показывать результат по каждому району.
- ЖКХ/налоги существуют, но их нужно превратить из таблицы ввода в полезный трекер с итогами, сроками и напоминаниями.
- Главная категория ведет в каталог, но не передает выбранный фильтр.
- Проблемы, сценарии и ситуации визуально и терминологически местами смешиваются.
- Иконки категорий слишком обобщенные и повторяются.
- Админ-аналитика смешивает реальные просмотры статей и представительские числа для пустых данных.
- WebView-shell показывает техническую диагностику и должен получить пользовательские offline/server-error экраны в стиле продукта.
