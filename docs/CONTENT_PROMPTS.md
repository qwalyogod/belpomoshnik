# Промпты для наполнения контента Белпомощника

Этот документ содержит готовые шаблоны research-промптов для генерации seed-данных проекта «Белпомощник» (информационная система для граждан Республики Беларусь). Промпты предназначены для передачи LLM-ассистенту (ChatGPT, Claude, локальная модель) с целью получить структурированный JSON, пригодный для импорта в backend.

Главная идея: разделить процесс на две роли. LLM собирает черновик строго по официальным источникам и выдаёт его в формате JSON. Редактор-человек верифицирует каждую запись, прежде чем JSON попадёт в `data/import/`. Никакие автоматически сгенерированные данные не публикуются без ручной проверки.

Документ покрывает шесть типов контента: жизненные сценарии, короткие проблемы, новости и закон-апдейты, источники, учреждения, чувствительные реестры. Каждый промпт написан как самодостаточный блок, который можно скопировать в чат с LLM без дополнительных пояснений.

Документ — это инструкция для редактора, а не код. Все сгенерированные артефакты хранятся вне репозитория до момента ручной верификации.

## 1. Общие правила для всех промптов

Эти правила действуют для любого промпта ниже, даже если в самом промпте они не повторены.

1. Источник обязателен. Каждый факт сопровождается тремя полями: `url`, `title`, `checked_at` (дата проверки, формат `YYYY-MM-DD`). Дата проверки не старше 6 месяцев от текущей даты.
2. Период поиска. Если запрос касается истории изменений, период источников ограничен 2022-2026.
3. Запрет на выдуманные ссылки. Если официальный источник не найден, в поле `url` писать строку `не найдено`, а не угадывать.
4. Разделение контента. Внутри одной записи факт, интерпретация и пользовательский шаг должны находиться в разных полях. Не смешивать.
5. Формат вывода. Только JSON, без markdown-обёрток, без пояснений до или после блока. JSON должен парситься стандартным парсером (`json.loads`).
6. Запрет на оценки. В тексте запрещены формулировки «это безопасно», «это законно», «это надёжно», «лучший вариант», «рекомендуется». Только нейтральное описание процедуры.
7. Несколько интерпретаций. Если в источнике есть расхождения, приводить все, помечая поле флагом `disputed: true`. Не выбирать одну из версий.
8. Только открытые источники. Закрытые реестры, платные базы, форумы, личные блоги не использовать как первичный источник. Допускаются только: `pravo.by`, `portal.gov.by`, сайты министерств и ведомств РБ, официальные реестры.
9. Язык. Текст карточек на русском. Термины на белорусском (`загс`, `звестка`) допустимы, если они взяты из официального источника.
10. Объём. Не раздувать запись. Если фактов мало, пустые массивы остаются пустыми массивами, не заполняются водой.

## 2. Жизненные сценарии

Промпт генерирует один жизненный сценарий (например, «Рождение ребёнка», «Открытие ИП», «Покупка квартиры в Минске»). Сценарий описывает длинную пользовательскую процедуру с этапами, шагами, документами, учреждениями и сроками.

```text
Ты — ассистент по подготовке справочного контента для информационной системы
«Белпомощник». Твоя задача — подготовить JSON-черновик одного жизненного сценария
для гражданина Республики Беларусь.

ВХОДНЫЕ ПАРАМЕТРЫ (замени подстановки):
- Тема: <СЮДА ВСТАВЬ ТЕМУ СЦЕНАРИЯ>
- Регион/город: <СЮДА ВСТАВЬ, например: Минск, Гомель, вся РБ>
- Год: <СЮДА ВСТАВЬ, например: 2026>
- Целевая персона: <родитель / ИП / пенсионер / наёмный работник / иное>
- Особые условия: <СЮДА ВСТАВЬ, например: первый ребёнок, без прописки, и т.п.>
  (если условий нет, напиши "нет")

ОБЯЗАТЕЛЬНЫЕ СЕКЦИИ В ОТВЕТЕ:
1. Метаданные: slug, title, summary (1-2 предложения, без оценочных суждений),
   target_persona, region, year, last_verified (YYYY-MM-DD).
2. Этапы (stages): массив объектов. У каждого этапа поля:
   - order (целое, начиная с 1)
   - title
   - goal (одно предложение, что достигается по итогу этапа)
   - steps (массив шагов)
   У каждого шага поля:
   - order
   - title
   - action (что конкретно делает пользователь, императивная форма)
   - notes (нюансы, "если X, то Y"; если их нет, пустой массив)
3. Документы (documents): массив объектов с полями:
   - title, type (заявление / паспорт / справка / квитанция / иное),
     issuing_authority (название учреждения или null), notes.
4. Учреждения (authorities): массив объектов с полями:
   - title, type (ЗАГС / поликлиника / МФЦ / налоговая / банк / иное),
     address (только типичный формат, без персональных данных),
     contacts (только публичные телефоны и сайты),
     working_hours.
   Если точный адрес или телефон в источнике отсутствует, использовать null
   и пометить disputed: true.
5. Сроки (deadlines): массив объектов с полями:
   - step_order (к какому шагу относится),
   - label (название срока, например "подать заявление в течение 30 дней"),
   - duration_days (целое число или null),
   - critical (boolean).
6. Стоимости (fees): массив объектов с полями:
   - step_order,
   - label,
   - amount (число в белорусских рублях или null),
   - note (например "госпошлина", "оплата услуги").
7. Частые ошибки (common_mistakes): массив строк, нейтральная формулировка,
   без "это глупо", "так делать нельзя".
8. Источники (sources): массив объектов с полями url, title, checked_at.
   Если официальный источник не найден — url = "не найдено".

В ФОРМАТЕ ВЫВОДА:
- Один JSON-объект (не массив).
- Без markdown-обёрток.
- Без пояснительного текста до или после.
- Кодировка UTF-8.

ЗАПРЕЩЕНО:
- Упоминать платные услуги, юридические консультации, адвокатов.
- Использовать слова "безопасно", "законно", "надёжно", "лучший", "оптимально".
- Ссылаться на неофициальные сайты, форумы, личные блоги.
- Добавлять шаги, документы или сроки, для которых нет официального источника.
- Домысливать сроки, если они явно не указаны в источнике.
- Использовать em-dash в тексте полей.

ТРЕБОВАНИЯ К ИСТОЧНИКАМ:
- Допустимы только pravo.by, portal.gov.by, сайты министерств и ведомств РБ.
- Минимум один источник на каждый этап с конкретной процедурой.
- Если процедура описана в нескольких актах, привести все.
- Дата проверки (checked_at) — не старше 6 месяцев от текущей даты.

ПРИМЕР ВХОДНЫХ ПАРАМЕТРОВ ДЛЯ ПРОВЕРКИ (не использовать в боевом запуске):
Тема: "Регистрация рождения ребёнка"
Регион: "вся РБ"
Год: "2026"
Целевая персона: "родитель"
Особые условия: "первый ребёнок, брак зарегистрирован"
```

## 3. Проблемы (короткие справочные карточки)

Промпт генерирует одну короткую карточку проблемы. Проблема — это ситуация, с которой пользователь столкнулся прямо сейчас и хочет узнать «что делать». Карточка компактнее сценария, без этапов.

```text
Ты — ассистент по подготовке справочного контента для информационной системы
«Белпомощник». Подготовь JSON-черновик карточки одной проблемы.

ВХОДНЫЕ ПАРАМЕТРЫ (замени подстановки):
- Тема: <СЮДА ВСТАВЬ, например: "Потерял паспорт">
- Срочность: <низкая / средняя / высокая / критическая>
- Контекст: <1-2 предложения, что произошло>
- Регион: <СЮДА ВСТАВЬ>

ОБЯЗАТЕЛЬНЫЕ ПОЛЯ В ОТВЕТE:
- slug (латиницей, kebab-case)
- title
- urgency (одно из: low / medium / high / critical)
- context_summary (1-2 предложения, нейтрально)
- what_to_do_now (массив строк, 3-5 пунктов, императивная форма,
  самое первое действие должно быть безопасным и доступным)
- step_by_step (массив объектов {order, action, notes})
- documents (массив объектов {title, type, notes})
- deadlines (массив объектов {label, duration_days, critical})
- where_to_go (массив объектов {authority_type, region, contacts_only_public: true})
- common_mistakes (массив строк)
- sources (массив {url, title, checked_at})

ТРЕБОВАНИЯ:
- where_to_go содержит ТОЛЬКО типы учреждений и публичные контакты.
  Никаких фамилий, имён, номеров кабинетов.
- what_to_do_now начинается с безопасного шага (позвонить, проверить,
  заблокировать). Не начинать с похода в учреждение, если есть риск
  финансовых или юридических последствий.
- Если срочность "критическая", первый пункт what_to_do_now должен быть
  связан с обращением в компетентный орган (полиция, скорая, МЧС).
- Все источники — официальные, проверены не позднее 6 месяцев назад.

ЗАПРЕЩЕНО:
- Слова "срочно немедленно бегите", паника, эмоциональные обороты.
- Юридические оценки ("это мошенничество", "вас обманули").
- Неподтверждённые сроки.
- Закрытые ссылки, личные кабинеты, требующие авторизации.

ФОРМАТ:
- Один JSON-объект, UTF-8, без markdown.
```

## 4. Новости и закон-апдейты

Раздел состоит из двух подтипов: редакционная новость и закон-апдейт. Закон-апдейт — структурированная запись об изменении нормативной базы, требует особого набора полей.

### 4.1. Редакционная новость

```text
Ты — ассистент по подготовке редакционных новостей для информационной системы
«Белпомощник». Подготовь JSON-черновик одной новости.

ВХОДНЫЕ ПАРАМЕТРЫ (замени подстановки):
- Тема: <СЮДА ВСТАВЬ>
- Категория: <общее / здравоохранение / образование / ЖКХ / транспорт / налоги / иное>
- Регион: <вся РБ / Минск / иное>
- Дата события: <YYYY-MM-DD>

ПОЛЯ:
- slug
- title
- category
- region
- event_date
- body (2-4 абзаца, нейтрально, без оценочных суждений)
- related_scenarios (массив slug-ов существующих сценариев или пустой)
- related_problems (массив slug-ов или пустой)
- sources (массив {url, title, checked_at})

ЗАПРЕЩЕНО:
- Использовать слова "сенсация", "шок", "эксклюзив", "важно знать".
- Указывать источники, не относящиеся к РБ.
- Домысливать причины события.

ТРЕБОВАНИЯ К ИСТОЧНИКАМ:
- Минимум один источник — официальный портал или пресс-служба.
- Дата проверки — не старше 6 месяцев.
```

### 4.2. Закон-апдейт

```text
Ты — ассистент по подготовке закон-апдейтов для информационной системы
«Белпомощник». Закон-апдейт — это короткая структурированная запись
о вступлении в силу изменений в нормативной базе Республики Беларусь.

ВХОДНЫЕ ПАРАМЕТРЫ (замени подстановки):
- Что изменилось: <СЮДА ВСТАВЬ КРАТКО>
- Нормативный акт: <название и номер, например: Закон РБ от 12.07.2024 N 17-З>
- Кого касается: <СЮДА ВСТАВЬ>
- С какого числа действует: <YYYY-MM-DD>
- Источник: <СЮДА ВСТАВЬ URL на pravo.by или официальный портал>

ОБЯЗАТЕЛЬНЫЕ ПОЛЯ:
- slug
- title (кратко, без эмоций)
- law_reference (полная ссылка на акт: тип, номер, дата принятия)
- effective_date (YYYY-MM-DD)
- audience (кто попадает под действие)
- what_changed (нейтральное описание сути изменения, 1-3 предложения)
- what_to_do (массив строк, что нужно сделать пользователю, императив)
- related_scenarios (массив slug-ов или [])
- related_problems (массив slug-ов или [])
- sources (массив {url, title, checked_at})
- verified_by (пустая строка, заполняется редактором)

ТРЕБОВАНИЯ К ИСТОЧНИКУ:
- ТОЛЬКО pravo.by или официальный портал министерства/ведомства.
- URL должен вести на конкретную страницу акта или на конкретную новость
  пресс-службы. Главная страница портала недопустима.
- Дата проверки — не старше 6 месяцев.

ЗАПРЕЩЕНО:
- Юридические оценки ("ужесточение", "послабление", "либерализация").
- Домыслы о последствиях, не указанные в акте.
- Повторение формулировок из рекламных материалов.
```

## 5. Источники

Промпт для поиска и валидации официального источника. Использовать, когда в уже существующей записи (сценарий, проблема, новость) обнаружена слабая или устаревшая ссылка.

```text
Ты — ассистент по валидации источников для информационной системы
«Белпомощник». Твоя задача — найти и подтвердить официальный источник РБ.

ВХОДНЫЕ ПАРАМЕТРЫ (замени подстановки):
- Что ищем: <СЮДА ВСТАВЬ, например: "порядок регистрации ИП">
- Где предполагается искать: <pravo.by / portal.gov.by / сайт министерства / иное>
- Для каких разделов подходит: <сценарии / проблемы / новости / закон-апдейты>

ОБЯЗАТЕЛЬНЫЕ ПОЛЯ В ОТВЕТE:
- title (название источника, как на сайте)
- source_type (одно из: law / government_portal / ministry / registry / tax_service / court)
- url (конкретная страница, не главная)
- short_description (1-2 предложения, что на этой странице)
- last_checked_at (YYYY-MM-DD, не старше 6 месяцев)
- applicable_to (массив строк: какие разделы контента могут на него ссылаться)
- alternative_urls (массив строк, запасные ссылки, или пустой массив)

ЗАПРЕЩЕНО:
- Выдумывать URL, если страница не найдена. В этом случае url = "не найдено".
- Использовать сокращатели ссылок, зеркала, кэшированные копии.
- Использовать форумы, Википедию, личные блоги как основной источник.

ПРАВИЛА:
- Если на странице несколько актов, выбрать наиболее релевантный и явно
  указать его в short_description.
- Если источник за период 2022-2026, зафиксировать точную дату публикации.
```

## 6. Учреждения

Промпт для карточки учреждения (ЗАГС, поликлиника, МФЦ, налоговая инспекция, банк-агент, и т.п.).

```text
Ты — ассистент по подготовке карточек учреждений для информационной системы
«Белпомощник».

ВХОДНЫЕ ПАРАМЕТРЫ (замени подстановки):
- Название: <СЮДА ВСТАВЬ, например: "Отдел ЗАГС администрации Октябрьского района г. Минска">
- Тип: <ЗАГС / поликлиника / МФЦ / налоговая / банк / иное>
- Регион/город: <СЮДА ВСТАВЬ>
- Что там можно сделать: <СЮДА ВСТАВЬ СПИСОК УСЛУГ через запятую>

ОБЯЗАТЕЛЬНЫЕ ПОЛЯ:
- title (полное официальное название)
- short_title (сокращённое, если есть в источнике)
- type (одно из: zags / clinic / mfc / tax / bank / police / minsy / mvd / pension / social / other)
- region
- city
- address (улица, дом; null если точный адрес в открытом источнике отсутствует)
- public_phones (массив строк, только публичные номера; null если закрыты)
- public_email (строка или null)
- website (URL официального сайта или null)
- working_hours (объект {mon, tue, wed, thu, fri, sat, sun} где значения — строки
  формата "9:00-18:00" или "выходной"; null если не опубликовано)
- services (массив строк, нейтральные формулировки без оценочных суждений)
- sources (массив {url, title, checked_at})

ЗАПРЕЩЕНО:
- Указывать фамилии, имена, отчества конкретных сотрудников.
- Указывать номера кабинетов, внутренние телефоны, личные мобильные.
- Указывать адреса частных лиц, частных консультантов.
- Заполнять поля выдуманными данными. Если данных нет, использовать null.

ПРАВИЛА:
- Если у учреждения несколько филиалов — это разные записи.
- Если в источнике только общий телефон "горячей линии", указать его.
- Дата проверки — не старше 6 месяцев.
```

## 7. Чувствительные реестры (экстремистский контент)

Самый строгий раздел. Промпт выдаёт только сухую фактологическую выписку из официального реестра. Никакой интерпретации, никаких характеристик, никаких имён сверх того, что есть в источнике.

```text
Ты — ассистент по подготовке структурированных выписок из официальных
реестров Республики Беларусь для информационной системы «Белпомощник».
Раздел «Чувствительные реестры» — самый строгий.

ВХОДНЫЕ ПАРАМЕТРЫ (замени подстановки):
- Какой реестр: <СЮДА ВСТАВЬ, например: "Реестр экстремистских материалов"
  или "Реестр запрещённых интернет-ресурсов">
- Конкретная запись (если известна): <СЮДА ВСТАВЬ ID, номер, дату включения>
- Контекст запроса: <СЮДА ВСТАВЬ, что ищем>

ОБЯЗАТЕЛЬНЫЕ ПОЛЯ В ОТВЕТE:
- record_id (номер/идентификатор записи в реестре, как в источнике)
- registry_name (полное название реестра)
- registry_authority (кто ведёт реестр: МВД / КГБ / Минюст / иное)
- registry_url (URL реестра, общий раздел)
- record_url (URL конкретной записи в реестре, если есть)
- included_at (YYYY-MM-DD, дата включения в реестр)
- title_as_in_registry (название ровно как в реестре, без пересказа)
- description_as_in_registry (если есть, цитатой)
- media_type (один из: text / video / audio / image / site / channel / other)
- last_checked_at (YYYY-MM-DD, не старше 6 месяцев)
- disclaimer (всегда строка:
  "Запись воспроизведена из открытого официального реестра. Перед
  принятием решений сверяйтесь с источником.")

ТРЕБОВАНИЯ К ИСТОЧНИКУ:
- ТОЛЬКО официальные реестры, опубликованные на сайтах МВД РБ, КГБ РБ,
  Министерства юстиции РБ.
- URL должен вести на конкретную запись. Главная страница реестра
  допустима только как registry_url, не как record_url.

ЗАПРЕЩЕНО КАТЕГОРИЧЕСКИ:
- Упоминать конкретные имена, каналы, соцсети, ссылки, которых нет
  в записи реестра. Каркас должен повторять ровно то, что опубликовано.
- Давать любые характеристики содержимого ("опасное", "вредное",
  "пропагандистское", "разжигающее" и т.п.).
- Делать выводы, обобщения, прогнозы.
- Приводить цитаты из материалов, упомянутых в реестре.
- Добавлять «контекст», «предысторию», «мнение экспертов».
- Использовать в качестве источника СМИ, блоги, телеграм-каналы,
  любые производные публикации.
- Использовать слова «запрещено», «запрещено к распространению» в полях
  title_as_in_registry и description_as_in_registry. Эти поля — точная
  цитата из реестра. Юридические формулировки добавляет редактор
  при публикации, а не LLM.

ФОРМАТ:
- Один JSON-объект, UTF-8, без markdown-обёрток.
- Все текстовые поля — строки. Если данных нет, null, не пустая строка.
- Поле disclaimer присутствует всегда, менять его содержимое запрещено.

ПОСЛЕ ПОЛУЧЕНИЯ ОТВЕТА:
- Редактор обязан вручную открыть record_url, сверить каждое поле
  с опубликованной записью и поставить отметку verified_by.
- Публикация без ручной верификации запрещена.
```

## 8. Как использовать промпты

Пошаговый сценарий работы редактора контента.

1. Открой нужный раздел документа (например, раздел 2 для сценариев).
2. Скопируй весь блок промпта в чат с LLM.
3. Замени подстановки `<СЮДА ВСТАВЬ>` на конкретные значения.
4. Запусти промпт. LLM выдаёт JSON-объект.
5. Проверь каждое поле: даты, URL, отсутствие запрещённых формулировок.
6. Сверь факты с указанными источниками, открой каждый URL.
7. Если найдены расхождения — переспроси LLM с указанием на ошибку или
   исправь вручную.
8. Сохрани JSON-файл в `data/import/<раздел>/<slug>.json` или в
   `data/import/<slug>.json` для плоских коллекций.
9. Когда накопится пачка файлов — выполни команду «импортируй данные из
   data/import» (см. раздел 9).

### Пример: сценарий «Открытие ИП в Минске, 2026»

- Открыть раздел 2.
- Скопировать промпт.
- Заменить: Тема = «Открытие ИП», Регион = «Минск», Год = «2026»,
  Целевая персона = «ИП», Особые условия = «первая регистрация».
- Запустить. Получить JSON.
- Сверить источники: `nalog.gov.by`, `portal.gov.by`, `pravo.by`.
- Сохранить в `data/import/scenarios/otkrytie_ip_2026.json`.
- Запросить импорт.

### Пример: закон-апдейт об изменении ставки

- Открыть раздел 4.2.
- Скопировать промпт.
- Заменить: Что изменилось = «ставка подоходного налога», Нормативный акт
  = «Закон РБ от 12.07.2025 N 17-З», Кого касается = «физические лица —
  резиденты РБ», С какого числа = «2026-01-01».
- Запустить. Получить JSON.
- Сверить URL с `pravo.by`.
- Сохранить в `data/import/news/<slug>.json`.
- Запросить импорт.

### Пример: запись в чувствительном реестре

- Открыть раздел 7.
- Скопировать промпт.
- Указать конкретную запись из реестра.
- Запустить. Получить JSON с полем disclaimer.
- Вручную открыть record_url, сверить каждое поле с опубликованной
  записью.
- Только после ручной верификации сохранить в
  `data/import/news/<slug>.json` и запросить импорт.

## 9. Импорт в backend

Где лежат файлы и как они попадают в базу.

Структура `data/import/`:

| Файл / папка | Что внутри | Источник промпта |
|--------------|-----------|-----------------|
| `regions.json` | Регионы/районы РБ (опционально) | раздел 5 / 6 (вручную) |
| `institutions.json` | Учреждения, плоский массив | раздел 6 |
| `problems.json` | Каталог проблем, плоский массив | раздел 3 |
| `scenarios.json` | Сценарии, плоский массив | раздел 2 |
| `news.json` | Новости и закон-апдейты, плоский массив | раздел 4 |
| `sensitive.json` | Записи чувствительных реестров, плоский массив | раздел 7 |

Правила:

- Формат файлов — UTF-8 JSON-массив `[ ... ]`. Без markdown-обёрток,
  без BOM.
- Имена файлов — `kebab-case`, латиницей.
- Каждая запись должна иметь уникальный `slug` внутри своего массива.
- Перед импортом редактор обязан пройти чек-лист:
  - каждый URL открыт и ведёт на релевантную страницу;
  - дата проверки (`checked_at`) не старше 6 месяцев;
  - в тексте нет слов «безопасно», «законно», «лучший»;
  - для чувствительных реестров: `verified_by` заполнено инициалами
    редактора.

Импорт:

- Положить файл (или набор файлов) в `data/import/`.
- Запустить в чате с ассистентом: «импортируй данные из data/import».
- Ассистент проверяет схему, валидирует `slug`, нормализует даты,
  вызывает backend-эндпоинт `POST /api/admin/import/...` для
  соответствующей сущности.
- После импорта получить отчёт: количество успешных и отклонённых
  записей, причины отклонений.
- Отклонённые записи вернуть в `data/import/` с пометкой `_rejected.json`
  и комментариями, исправить, повторить импорт.

Связанные документы:

- `data/import/README.md` — описание содержимого каталога.
- `docs/SCENARIO_BACKEND.md` — структура backend-моделей, в которые
  импортируются записи.
- `docs/SOURCES.md` — перечень официальных источников, разрешённых
  для использования.
- `docs/SECURITY_POLICY.md` — ограничения для чувствительных разделов.
