# Экраны приложения

## Общая iPhone-адаптация

- Статус: реализовано для текущего мобильного MVP.
- Все основные мобильные экраны выводятся через общий SafeArea layout в `src/components/layout.py`.
- Верхний safe area защищает контент от status bar, notch и Dynamic Island.
- Нижняя навигация находится в отдельной safe area и не должна попадать в home indicator-зону.
- Длинные экраны скроллятся в общем контейнере, а последний блок получает нижний отступ над навигацией.
- Проверяемые ширины: 375 px, 393 px, 430 px.
- Android и планшеты планируются отдельным этапом.

## LoginPage

- Назначение: экран входа пользователя.
- Статус: реализовано как локальный MVP без backend/JWT.
- Основные элементы: поля телефона/email и пароля, кнопка входа, ссылка на регистрацию.
- Данные: локальный демо-пользователь и пользователи, сохранённые в `data/app_state.json`.
- Действия: войти, перейти к регистрации. Восстановление доступа планируется после backend-авторизации.

## RegisterPage

- Назначение: регистрация нового пользователя.
- Статус: реализовано как локальный MVP без backend/JWT.
- Основные элементы: поля имени, email, города, региона, пароля, подтверждения пароля.
- Данные: локальные данные аккаунта с сохранением в `data/app_state.json`.
- Действия: создать аккаунт, перейти ко входу.

## OnboardingPage

- Назначение: краткое знакомство при первом запуске.
- Статус: реализовано в MVP.
- Основные элементы: три шага — выбрать сценарий, получить пошаговый план, отмечать задачи и следить за прогрессом.
- Данные: состояние `onboarding_completed` сохраняется в `data/app_state.json`.
- Действия: нажать «Начать» или «Пропустить»; после этого onboarding больше не показывается.

## HomePage

- Назначение: стартовый экран приложения.
- Статус: реализовано как пользовательский дашборд MVP на локальных данных.
- Основные элементы: приветствие, поиск, быстрые переходы, статистика ситуаций, активные ситуации, ближайшие задачи, просроченные задачи, документы к подготовке, напоминания, категории, популярные проблемы.
- Данные: локальные ситуации, задачи и уведомления из `data/app_state.json`, моковые категории и популярные проблемы.
- Действия: перейти к жизненным сценариям, перейти к моим ситуациям, открыть активную ситуацию, открыть проблему, выбрать категорию, ввести запрос и перейти в каталог с фильтрацией.

## ProblemsPage

- Назначение: каталог жизненных проблем.
- Статус: реализовано в MVP на моковых данных.
- Основные элементы: список проблем, карточки проблем, поиск, фильтры категорий.
- Данные: моковые проблемы из `src/data/mock_data.py`.
- Действия: искать по названию, описанию и категории; фильтровать по категории; открыть карточку проблемы.

## ProblemDetailPage

- Назначение: подробная карточка проблемы.
- Статус: реализовано в MVP; для «Рождение ребёнка» добавлена частичная загрузка структуры шагов из public API (с fallback на моки).
- Основные элементы: название проблемы, категория, описание, «Что делать прямо сейчас», «Пошаговый план», чекбоксы, прогресс, «Документы», «Сроки», «Куда обращаться», «Частые ошибки», блок «Проверьте себя».
- Данные: моковая карточка проблемы, частичный API override для «Рождение ребёнка» и моковый микро-тест.
- Действия: отметить шаги, увидеть прогресс, сохранить карточку локально, создать персональный план, пройти микро-тест.

## SituationsPage

- Назначение: список личных ситуаций пользователя.
- Статус: реализовано в локальном интерактивном MVP.
- Основные элементы: карточки ситуаций, статус, прогресс, количество задач, фильтры «Все», «Активные», «Завершённые», кнопка добавления ситуации, кнопка выбора готового сценария.
- Данные: моковые и локально добавленные ситуации.
- Действия: открыть ситуацию, добавить новую ситуацию, редактировать название и статус, удалить ситуацию с подтверждением; прогресс считается автоматически по задачам.

## ScenarioTemplatesPage

- Назначение: каталог готовых жизненных сценариев.
- Статус: реализовано в MVP на моковых данных.
- Основные элементы: поиск, фильтры категорий, карточки сценариев, категория, описание, примерный срок, сложность, избранное, пустое состояние.
- Данные: `scenario_templates` из `src/data/mock_data.py`: «Рождение ребёнка», «Потеря паспорта», «Расторжение брака», «Открытие ИП», «Переезд и регистрация».
- Действия: искать сценарий, фильтровать по категории, добавить или убрать из избранного, открыть детальную страницу сценария.

## ScenarioDetailPage

- Назначение: детальный просмотр шаблона жизненного сценария.
- Статус: реализовано в MVP на моковых данных.
- Основные элементы: краткое описание, этапы, задачи, зависимости шагов, типы учреждений для шагов, документы, организации, официальные источники, связанные сценарии, кнопка «Создать мою ситуацию».
- Данные: выбранный шаблон из `SCENARIO_TEMPLATES`; каждый шаблон содержит этапы, задачи, `depends_on`, `institution_types`, документы, организации, источники и при наличии связанные сценарии. Основной демо-шаблон «Рождение ребёнка» содержит 7 этапов.
- Действия: создать локальную ситуацию из шаблона, открыть официальный источник, вернуться к списку сценариев. Если активная ситуация из этого шаблона уже есть, приложение предлагает открыть существующую или создать ещё одну.

## SituationDetailPage

- Назначение: детальный просмотр персонального плана.
- Статус: реализовано в локальном интерактивном MVP.
- Основные элементы: список шагов, группировка по этапам, фильтры задач, прогресс, количество выполненных задач, документы к подготовке, задачи, сроки, блокировка зависимых задач, блок «Куда обращаться».
- Данные: локальные задачи, привязанные к конкретной ситуации, моковые ситуации, документы задач из шаблона и ручных задач, текстовые сроки, `due_date`, `depends_on`, `institution_types`, подобранные учреждения из `INSTITUTIONS`.
- Действия: отметить доступную задачу выполненной, увидеть причину блокировки зависимой задачи, посмотреть подходящие учреждения для шага, фильтровать задачи, добавить свою задачу с датой срока и документами, редактировать задачу, дату срока и документы, удалить задачу с подтверждением, посмотреть уникальные документы из задач, сохранить прогресс текущей ситуации, редактировать или удалить текущую ситуацию.

## DocumentsPage

- Назначение: список важных документов пользователя.
- Статус: реализовано в локальном интерактивном MVP; начат спринт личных документов и демонстрационной безопасности.
- Основные элементы: карточки документов, тип документа, статус, серия/номер, срок действия, организация выдачи, комментарий, переключатель «Скрывать личные данные», кнопка «Показать данные», FilePicker для выбора скана, кнопка открытия скана при наличии пути.
- Данные: `personal_documents` из `data/app_state.json`, моковые начальные документы из `src/data/mock_data.py`, настройки приватности из `privacy_settings`.
- Действия: добавить личный документ, редактировать поля документа, удалить документ с подтверждением, включить/выключить маскирование реквизитов, временно показать номер документа, увидеть предупреждение перед просмотром скана.
- Ограничение: production-шифрование и серверная политика хранения персональных данных планируются следующими этапами.

## NotificationsPage

## LegalUpdatesPage

- Основные элементы: блок «Важное для вас», список обновлений, поиск, категории, дата, приоритет, краткое объяснение, пустое состояние.

## LawDetailPage

- Статус: реализовано в локальном интерактивном MVP.
- Основные элементы: название, категория, дата вступления, кого касается, краткое описание, список изменений, что сделать пользователю, связанные сценарии, официальный источник.
- Данные: локальные `law_updates` и `law_detail` с fallback на моковые данные.

## ProfilePage

- Назначение: профиль и настройки пользователя.
- Статус: реализовано в локальном интерактивном MVP.
- Основные элементы: блок пользователя, интересы, достижения, личные данные, настройки, переключатель режима обучения, блок демо-режима.
- Данные: локальный пользователь, настройки, избранные сценарии и моковые достижения.

- Данные: статичный текст проекта.
- Действия: вернуться в профиль.

## LearningProgressPage

- Статус: реализовано в MVP на моковых данных.
- Основные элементы: заголовок «Обучение», пояснение, статистика, прогресс по категориям, достижения.
- Данные: `learning_stats`, `learning_categories`, `achievements` из `src/data/mock_data.py`.
- Действия: посмотреть прогресс и достижения. Реальное сохранение результатов планируется.

## AdminPage

- Назначение: админ-панель для управления справочниками и контентом.
- Статус: реализована MVP-структура + рабочая API-интеграция конструктора сценариев.
- Основные элементы:
  - блок «Роли и доступ» с режимами `Гражданин`, `Редактор контента`, `Админ площадки`;
  - список доступных разделов и прав выбранной роли;
  - демо-пользователи и последние действия аудита для роли «Админ площадки»;
  - статус backend API и кнопка синхронизации;
  - формы создания проблем и сценариев;
  - список проблем и сценариев с публикацией/возвратом в draft;
  - форма создания закон-апдейта;
  - список закон-апдейтов редактора со статусами, приоритетом, связанными сценариями, редактированием и удалением;
  - форма добавления официального источника;
  - список официальных источников со статусом проверки, типом, датой проверки, редактированием и удалением;
  - конструктор сценария: выбор сценария, формы добавления этапа и шага;
  - список этапов и шагов выбранного сценария с переключением обязательности.
- Данные: локальные `admin_roles`, `admin_users`, `admin_audit_logs`, `law_updates`, `law_detail`, `official_sources` + backend API/БД для сценариев.
- Действия:
  - переключить демонстрационный режим роли;
  - через API: загрузка списков проблем/сценариев/документов/сроков/организаций;
  - создание проблем, сценариев, этапов и шагов;
  - создание локального закон-апдейта;
  - редактирование локального закон-апдейта: категория, статус, приоритет, кого касается, что изменилось, что сделать пользователю, источник и связанные сценарии;
  - публикация/возврат в черновик локального закон-апдейта;
  - удаление локального закон-апдейта с записью в демо-аудит;
  - создание, редактирование, удаление и смена статуса официального источника;
  - привязка документов к шагу при создании;
  - переключение обязательности этапов и шагов;
  - TODO: добавить настоящий backend RBAC, управление пользователями, CRUD источников, зависимости, связь закон-апдейтов с проблемами и полноценное редактирование всех сущностей.
