# Отчёт о состоянии проекта «Белпомощник» по ТЗ с точки зрения пользователя

Дата отчёта: 2026-06-02.

Источник требований: `ТЗ Абрамович.docx`.

Текущий статус проекта: демонстрационная beta-версия с рабочим Flet/Python-интерфейсом, локальным состоянием, частичной backend-основой и web-preview для проверки.

## 1. Краткий вывод

Проект уже закрывает главную пользовательскую идею ТЗ: человек заходит в приложение, выбирает жизненную ситуацию, получает пошаговый план, видит задачи, документы, сроки, организации, уведомления и прогресс. Это уже можно показывать как работающий MVP.

С точки зрения пользователя приложение выглядит не как набор отдельных справочных страниц, а как личный помощник: есть сценарии, личные ситуации, дашборд, документы, уведомления, профиль, обучение, ЖКХ/налоги и закон-апдейты.

Но до полного закрытия ТЗ ещё остаются системные части: настоящая backend-авторизация, роли и права, полноценная админка редактора, PostgreSQL, SMTP/email, production-безопасность документов, реальная актуализация закон-апдейтов и проверенные официальные данные учреждений.

## 2. Важное отличие от исходного ТЗ

В ТЗ система описана как web-система: адаптивный сайт, frontend на HTML/CSS/JS, backend на Node.js Express или PHP, рекомендуемая БД PostgreSQL.

Фактический проект сейчас реализован иначе:

- основной клиент: Flet/Python;
- проверка сейчас ведётся через web-режим Flet;
- локальное состояние: `data/app_state.json`;
- backend-основа: FastAPI + SQLAlchemy + SQLite MVP;
- PostgreSQL, JWT/RBAC и production backend пока не внедрены.

Это не ломает пользовательскую идею ТЗ, потому что основная логика «проблема → сценарий → задачи → документы → прогресс» уже реализована. Но в дипломной пояснительной записке нужно явно объяснить, что технология интерфейса была изменена на Flet/Python для кроссплатформенного мобильного приложения и демонстрационного web-preview.

## 3. Пользовательский взгляд: что человек может сделать уже сейчас

### 3.1. Первый вход

Пользователь может открыть приложение, пройти onboarding, войти через локальный демо-аккаунт или зарегистрироваться локально.

Что работает:

- экран входа;
- экран регистрации;
- onboarding первого запуска;
- локальное сохранение профиля;
- демо-вход `ivan@example.by / 123456`.

Ограничение:

- это локальный MVP, а не настоящая серверная авторизация;
- пароли и роли не защищены production-механизмами.

### 3.2. Главная страница

Пользователь видит главный дашборд, а не пустую стартовую страницу.

Что есть на главной:

- поиск проблемы;
- быстрые переходы;
- активные ситуации;
- ближайшие задачи;
- просроченные задачи;
- документы к подготовке;
- напоминания;
- рекомендации начать со сценария.

Пользовательская ценность: человек сразу понимает, что сейчас важно сделать, и не обязан вручную искать все разделы.

Ограничение:

- поиск пока простой, по локальным данным;
- нет интеллектуального разбора свободного текста.

### 3.3. Жизненные сценарии

Пользователь может открыть каталог жизненных сценариев и выбрать готовый маршрут.

Реализованные сценарии:

- «Рождение ребёнка»;
- «Потеря паспорта»;
- «Расторжение брака»;
- «Открытие ИП»;
- «Переезд и регистрация».

Что видит пользователь:

- название сценария;
- категорию;
- краткое описание;
- примерный срок;
- сложность;
- этапы;
- задачи;
- документы;
- организации;
- источники;
- связанные сценарии;
- кнопку создания личной ситуации.

Пользовательская ценность: приложение работает не как обычная статья, а как готовый маршрут действий.

Ограничение:

- юридические тексты и источники пока демонстрационные;
- перед защитой и реальным использованием нужно сверить данные с официальными ресурсами.

### 3.4. Создание личной ситуации

Пользователь может нажать «Создать мою ситуацию» в сценарии. После этого приложение создаёт личный план.

Что работает:

- ситуация создаётся локально;
- задачи автоматически добавляются из шаблона;
- задачи привязываются к конкретной ситуации;
- документы подтягиваются из задач;
- прогресс считается автоматически;
- состояние сохраняется в `data/app_state.json`.

Пользовательская ценность: пользователь получает не просто информацию, а персональный чек-лист.

Ограничение:

- персонализация по региону/статусу пользователя пока частичная;
- нет серверного сохранения ситуаций.

### 3.5. Мои ситуации

Пользователь видит список своих активных и завершённых ситуаций.

Что работает:

- список ситуаций;
- фильтры;
- статус;
- прогресс;
- количество задач;
- редактирование ситуации;
- удаление ситуации;
- переход к деталям.

Внутри ситуации пользователь может:

- видеть этапы;
- видеть задачи;
- отмечать задачи выполненными;
- видеть заблокированные задачи;
- видеть причину блокировки;
- видеть документы к задачам;
- видеть сроки;
- добавлять, редактировать и удалять задачи.

Пользовательская ценность: приложение ведёт человека по шагам и не даёт перескочить через зависимые действия.

Ограничение:

- зависимости уже работают локально, но редактор ещё не может полноценно управлять всеми зависимостями через админку;
- нет синхронизации с сервером.

### 3.6. Мои документы

Пользователь может вести локальный список личных документов.

Что работает:

- добавление документа;
- редактирование документа;
- удаление документа;
- тип документа;
- серия/номер;
- дата выдачи;
- срок действия;
- организация выдачи;
- комментарий;
- файл/скан;
- автоматический статус: активен / скоро истекает / истёк;
- маскирование номера;
- кнопка показа реквизитов;
- предупреждение перед просмотром скана;
- импорт PDF как личного документа;
- экспорт PDF-отчёта по документам.

Пользовательская ценность: пользователь может контролировать паспорт, медкнижку и другие документы, видеть сроки и получать напоминания.

Ограничение:

- файлы хранятся локально;
- нет настоящего шифрования;
- нет серверной политики хранения персональных данных;
- PDF-экспорт демонстрационный.

### 3.7. Уведомления

Пользователь видит центр уведомлений.

Что работает:

- локальные уведомления;
- отметка уведомления прочитанным;
- отметка всех прочитанными;
- уведомления по задачам;
- уведомления по документам;
- уведомления по ЖКХ;
- уведомления по налогам;
- уведомления по важным закон-апдейтам;
- демо-email уведомление.

Пользовательская ценность: пользователь не должен сам помнить все сроки.

Ограничение:

- email реально не отправляется через SMTP;
- push-уведомлений нет;
- cron-планировщика нет;
- уведомления создаются локально при работе приложения.

### 3.8. Закон-апдейты

Пользователь может открыть ленту изменений законодательства.

Что работает:

- список обновлений;
- поиск;
- фильтры категорий;
- блок «Важное для вас»;
- карточка изменения;
- описание «что изменилось»;
- «кого касается»;
- «что сделать»;
- источник;
- связь со сценариями.

Пользовательская ценность: пользователь получает не поток новостей, а более понятные изменения, связанные с его жизненными ситуациями.

Ограничение:

- нет автоматической загрузки законов с официальных порталов;
- нет финальной юридической верификации;
- релевантность по профилю частичная.

### 3.9. ЖКХ и налоги

В проекте уже есть локальные трекеры ЖКХ и налогов.

ЖКХ-трекер позволяет:

- добавить лицевой счёт;
- хранить адрес;
- вести журнал платежей;
- отмечать статус оплаты;
- видеть просрочки и ожидающие платежи.

Налоговый трекер позволяет:

- добавить налоговое обязательство;
- указать тип пользователя: физлицо / ИП;
- указать срок;
- указать сумму;
- прикрепить квитанцию;
- отметить оплату.

Пользовательская ценность: приложение начинает закрывать бытовые обязательства, которые прямо указаны в ТЗ.

Ограничение:

- нет интеграции с госреестрами, налоговой или РСЦ;
- всё вводится вручную;
- уведомления и связи с профилем ещё требуют доработки.

### 3.10. Профиль и персонализация

Пользователь может заполнить профиль и настройки.

Что есть:

- имя;
- email;
- регион и город;
- интересы;
- признаки профиля: дети, авто, жильё и т.п.;
- статус занятости;
- настройки крупного шрифта;
- высокий контраст;
- режим обучения;
- избранные сценарии;
- достижения;
- журнал действий;
- сброс демо-данных;
- экран «О приложении».

Пользовательская ценность: профиль уже начинает влиять на рекомендации, сценарии, учреждения и закон-апдейты.

Ограничение:

- персонализация пока не является полноценным rule engine;
- настройки доступности ещё не полностью применяются ко всем экранам;
- нет серверного профиля.

### 3.11. Обучение

Пользователь может включить режим обучения и открыть экран прогресса.

Что есть:

- экран обучения;
- статистика;
- достижения;
- прогресс по категориям;
- микро-тесты.

Пользовательская ценность: приложение не только подсказывает, но и помогает закрепить важные бытовые знания.

Ограничение:

- обучение вспомогательное;
- нет полноценной системы курсов и сохранения результатов на сервере.

## 5. Критерии приёмки MVP из ТЗ

| № | Критерий из ТЗ | Статус | Комментарий |
|---|---|---|---|
| 1 | Поиск проблемы и просмотр карточки с пошаговым планом | Выполнено | Есть поиск, каталог проблем и карточка проблемы. |
| 2 | Пользователь может сохранить план и отмечать шаги выполненными | Выполнено в MVP | Через создание личной ситуации и чек-листы задач. |
| 3 | Life event: запуск ситуации → персональные задачи с дедлайнами | Выполнено | Ситуация создаётся из шаблона, задачи добавляются, дедлайны учитываются. |
| 4 | Закон-апдейты: лента + фильтрация «для меня» | Частично выполнено | Лента, фильтры и «Важное для вас» есть, но без реального источника и server-side персонализации. |
| 5 | Уведомления web + email минимум по двум сценариям | Частично выполнено | Web-уведомления есть; email пока демо-уведомление без SMTP. |
| 6 | Документы: паспорт + медкнижка, напоминания об истечении срока | Выполнено локально | Документы и статусы сроков есть, уведомления генерируются локально. |
| 7 | Админка: редактор может создавать проблемы/шаги и закон-апдейты | Частично выполнено | MVP-админка есть, но CRUD ещё не полный по всем сущностям. |

## 6. Что реализовано хорошо

### Для пользователя

- Приложение уже даёт понятный маршрут действий.
- Основной сценарий «Рождение ребёнка» можно показать как полноценный пример.
- Ситуации, задачи, документы и прогресс связаны между собой.
- Дашборд показывает ближайшие действия и просрочки.
- Документы можно вести локально и контролировать сроки.
- Уведомления уже не просто статичные, часть формируется из задач/документов/трекеров.
- Есть избранные сценарии, обучение, профиль и демо-сброс.

### Для диплома

- Видна архитектура будущей системы.
- Есть разделение на страницы, компоненты, сервисы, данные и backend.
- Есть локальное состояние с backup.
- Есть backend-основа для сценариев и админки.
- Есть документация, дорожные карты и отчёты по соответствию ТЗ.

| Доступность | Переключатели и крупный UI | Применить крупный шрифт/контраст глобально |

## 8. Что пока не реализовано

- Production PostgreSQL.
- Полный backend вместо локального JSON для пользовательских данных.
- Настоящая регистрация и авторизация.
- JWT / сессии production-уровня.
- Хеширование паролей bcrypt/argon2.
- RBAC для гражданина, редактора и администратора.
- HTTPS и защита от brute force.
- Реальный SMTP/email.
- Cron-планировщик уведомлений.
- Реальные push-уведомления.
- Реальное серверное хранение документов и сканов.
- Шифрование файлов документов.
- Полноценный аудит действий админов/редакторов на backend.
- Полноценный CRUD всех сущностей админки.
- Drag-and-drop порядка шагов и этапов.
- Управление шаблонами заявлений/документов как отдельной библиотекой.
- Автоматическая загрузка изменений законодательства из официальных источников.
- Финальная юридическая проверка сценариев и источников.
- Production-сборка под iOS/Android.

## 9. Пользовательский сценарий, который уже можно показывать

1. Пользователь открывает приложение.
2. Проходит onboarding или сразу попадает на главную.
3. Видит дашборд: ситуации, задачи, документы, напоминания.
4. Переходит в «Сценарии».
5. Открывает «Рождение ребёнка».
6. Видит этапы, задачи, документы, учреждения и источники.
7. Нажимает «Создать мою ситуацию».
8. Переходит в «Мои ситуации».
9. Открывает созданную ситуацию.
10. Отмечает доступные задачи выполненными.
11. Видит изменение прогресса.
12. Видит заблокированные задачи, если предыдущие шаги не выполнены.
13. Возвращается на главную и видит обновлённый дашборд.
14. Открывает документы и видит сроки действия.
15. Открывает уведомления и видит напоминания.
16. Открывает профиль, избранное, обучение и настройки.

Этот сценарий соответствует главной идее ТЗ: пользователь проходит жизненную ситуацию не как статью, а как управляемый план.

## 10. Что нужно сделать следующим этапом

### Приоритет 1. Довести внутреннюю логику пользователя

- Связать ЖКХ и налоги с главным дашбордом ещё плотнее.
- Довести уведомления по документам, налогам и ЖКХ.
- Сделать понятные пользовательские настройки напоминаний.
- Применить настройки доступности глобально.

### Приоритет 2. Довести админку по ТЗ

- Полный CRUD сценариев.
- Редактирование этапов и шагов.
- Удаление этапов и шагов.
- Управление порядком шагов.
- Управление зависимостями.
- Управление связанными сценариями.
- CRUD учреждений.
- CRUD шаблонов документов/заявлений.
- Полный журнал действий редактора.

### Приоритет 3. Production backend

- Выбрать финальный backend-курс: развивать текущий FastAPI или переходить на стек из ТЗ.
- Перейти с SQLite/JSON к PostgreSQL.
- Реализовать auth endpoints.
- Реализовать JWT/RBAC.
- Перевести мобильный клиент на API.
- Добавить server-side уведомления.
- Добавить SMTP/email.

### Приоритет 4. Безопасность

- Хеширование паролей.
- HTTPS.
- RBAC.
- Ограничение попыток входа.
- Политика хранения персональных данных.
- Шифрование документов.
- Аудит действий.
- Резервное копирование базы.

### Приоритет 5. Юридическая достоверность

- Проверить сценарии по официальным источникам.
- Уточнить сроки, документы, госпошлины и учреждения.
- Добавить дату последней проверки источников.
- Добавить ответственного редактора/статус проверки.

## 11. Итоговая оценка

Проект уже готов как демонстрационный MVP: пользователь может зайти, выбрать жизненный сценарий, создать личную ситуацию, выполнять задачи, видеть документы, прогресс и уведомления.

С точки зрения ТЗ лучше всего закрыты:

- главный модуль проблемы и сценариев;
- Life Events;
- личные ситуации;
- чек-листы;
- документы;
- дашборд;
- локальные уведомления;
- закон-апдейты в MVP-виде;
- профиль и персонализация в базовом виде;
- админ-панель как концепция и частичный инструмент редактора.

Частично закрыты:

- роли;
- backend;
- API;
- юридическая актуализация;
- email;
- безопасность;
- админка полного уровня.

Не закрыты production-части:

- PostgreSQL;
- JWT/RBAC;
- SMTP;
- cron;
- production security;
- финальное серверное хранение данных.

Главный вывод: текущая версия уже показывает правильное направление и пользовательскую ценность проекта. Следующий этап должен быть не про визуальную полировку, а про системное закрытие ТЗ: роли, админка, персонализация, уведомления, безопасность и backend.
