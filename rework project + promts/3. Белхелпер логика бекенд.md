Сделай класс:

`GroqProvider`

* использует env:

3. Добавь безопасное хранение пользовательского Groq key.

Нужно дать пользователю возможность вставить свой Groq API key в настройках, но безопасно.

Нельзя:

Модель:

* `id`
* `user_id`
* `provider` = `groq`
* `encrypted_api_key`
* `masked_key`, например `gsk_...abcd`
* `model`
* `is_enabled`
* `last_checked_at`
* `created_at`
* `updated_at`

Шифрование:

* используй `cryptography.Fernet`;
* ключ шифрования бери из env `BELPOMOSHNIK_AI_KEYS_MASTER_KEY`;
* добавь в `.env.example` команду генерации:
  `python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"`
* если пользовательский ключ есть, backend расшифровывает его только в момент запроса к Groq;
* никогда не возвращай полный ключ на frontend.

Endpoint'ы:

* `GET /api/user/ai-settings`

  * возвращает:

    * `server_provider_available`
    * `user_key_configured`
    * `masked_key`
    * `model`
    * `effective_mode`: `server_key` / `user_key` / `local_fallback`
* `PUT /api/user/ai-settings/groq-key`

  * принимает key и model;
  * шифрует key;
  * сохраняет masked_key;
  * опционально проверяет key коротким тестовым запросом;
* `DELETE /api/user/ai-settings/groq-key`

  * удаляет ключ;
* `POST /api/user/ai-settings/test`

  * проверяет доступность AI и возвращает дружелюбный статус.

Поведение:

* если серверный `GROQ_API_KEY` есть, обычные пользователи могут пользоваться AI без своего ключа;
* если серверного ключа нет и пользовательского ключа нет, чат работает в local fallback и frontend показывает: “Чтобы включить полноценный AI, добавьте Groq API key в настройках”;

5. Переделай RAG.

Нужно, чтобы нейросеть знала всё, что есть на сайте, но без галлюцинаций.

Сделай нормальную функцию `collect_rag_context(db, user, message)`.

Она должна искать по:

* опубликованным проблемам;
* опубликованным сценариям;
* этапам сценариев;
* шагам сценариев;
* документам, связанным со сценариями;
* учреждениям;
* статьям/новостям;
* правовым обновлениям;
* источникам;
* экстремистскому реестру, если есть опубликованные записи;
* разделам сайта;
* пользовательскому профилю в безопасном виде: регион, город, интересы, флаги `has_car`, `has_children`, `owns_property`, `is_renter`.

Не передавай LLM персональные документы пользователя, номера документов, сканы, адреса целиком и другую чувствительную информацию.

Для поиска:

* на первом этапе можно сделать гибридный keyword/BM25-like поиск без embeddings;
* нормализуй русский текст: lower, `ё -> е`, очистка пунктуации;
* добавь синонимы:

  * паспорт: документ, удостоверение, ID-карта, потерял паспорт;
  * ЖКХ: коммуналка, лицевой счёт, показания, квитанция, оплата;
  * налоги: МНС, налоговая, транспортный налог, недвижимость, земля;
  * ребёнок: ребенок, рождение, пособие, сад;
  * авто: машина, водительское, права, ГАИ;
  * работа: трудоустройство, увольнение, больничный;
* учитывай регион/город: если учреждение из города пользователя, повышай score;
* учитывай тип запроса:

  * “куда идти” → выше учреждения;
  * “что делать” → выше проблемы и сценарии;
  * “добавь/хочу оформить” → выше сценарии;
  * “новость/изменения/закон” → выше статьи и law updates.

Результат RAG должен быть списком `RetrievedEntity`:

* `kind`
* `id`
* `title`
* `route`
* `snippet`
* `score`
* `source_label`
* `can_create_situation`
* `metadata`

Важно:

* route должен формироваться server-side;
* LLM не должен сам придумывать route;
* если `kind=scenario`, route должен быть `/scenarios/{id}`;
* если `kind=problem`, route должен быть `/problem-detail/{id}`;
* если `kind=news`, route должен быть `/news?article={id}`;
* если `kind=law`, route должен быть `/law-detail/{id}`;
* для `authority` не используй фейковый `/scenarios/{authority_id}`.
  Если отдельной страницы учреждения нет, сделай route:

  * `/catalog?authority={id}` если можно быстро поддержать;
  * или `/catalog`;
  * или добавь новую страницу/модалку учреждения.
    Главное — не вести на неправильный сценарий.

6. Структурированный ответ ассистента.

Сделай response schema:

```json
{
  "text": "ответ пользователю",
  "links": [
    {
      "title": "string",
      "path": "string",
      "kind": "scenario|problem|authority|document|news|law|section",
      "snippet": "string"
    }
  ],
  "actions": [
    {
      "type": "create_user_situation",
      "label": "Добавить в мои ситуации",
      "scenario_id": "string",
      "title": "string"
    }
  ],
  "sources": [
    {
      "kind": "scenario",
      "title": "string",
      "path": "string"
    }
  ],
  "confidence": "high|medium|low",
  "needs_clarification": false
}
```

Правила:

* LLM может предлагать только actions на основе server-side retrieved entities;
* если LLM вернул лишние ссылки/actions, backend должен их отфильтровать;
* максимум 3 ссылки;
* максимум 2 actions;
* если confidence low, ответ должен попросить уточнить проблему и предложить каталог.
