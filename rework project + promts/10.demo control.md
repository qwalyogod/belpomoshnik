Ты работаешь с моим дипломным проектом «Белпомощник».

Задача: реализовать полноценный скрытый Control Center — релизный центр управления платформой, который не является демо-функцией и должен попасть в production-сборку. Это не обычная админ-панель и не раздел для редакторов/администраторов. Это скрытая системная панель владельца проекта, доступ к которой открывается только через специальную команду в консоли и дополнительный пароль.

Важно:

* не использовать название Demo Control Center;
* не использовать слово demo для основной панели, API, компонентов и моделей;
* правильное название: `Control Center`, `System Control Center`, `Owner Control Center` или “Центр управления платформой”;
* панель должна попасть в релиз;
* панель не должна отображаться в меню, админке, списках ролей или интерфейсе управления пользователями;
* не нужно создавать отдельную роль, которую можно назначить пользователю через админку;
* доступ считается “секретным”: знает команду + знает пароль;
* команда в консоли только открывает интерфейс, но пароль должен проверяться на backend;
* пароль нельзя хранить открытым текстом во frontend bundle;
* все действия Control Center должны логироваться.

Контекст:
Control Center нужен для централизованного управления всей платформой. Через него владелец проекта должен иметь возможность:

* включать техническое обслуживание;
* включать режим только чтения;
* отправлять уведомления пользователям;
* включать системный баннер;
* менять брендинг;
* менять порядок разделов в меню отдельно для desktop/tablet/mobile;
* включать и отключать функции проекта;
* запускать тестовые системные сценарии;
* создавать тестовые данные для проверки;
* смотреть live status проекта;
* видеть журнал действий.

Главная логика доступа:

1. Control Center нигде не отображается в обычном интерфейсе.
2. В обычной админке нельзя назначить доступ к Control Center.
3. В списке ролей не должно быть видимой роли “super admin”, “owner” или похожей.
4. Панель открывается только через команду в консоли:

```js
belpomoshnikControl()
```

5. После команды открывается скрытый экран входа в Control Center.
6. Экран требует пароль.
7. Пароль для текущей реализации:

```text
x20b01
```

8. Пароль проверяется backend'ом.
9. После правильного пароля backend выдаёт временный control-token.
10. Все опасные действия требуют этот control-token.
11. Если пароль неверный — доступ запрещается.
12. Если слишком много неверных попыток — временная блокировка входа в Control Center.

Env-настройки:

```env
CONTROL_CENTER_ENABLED=true
CONTROL_CENTER_PASSWORD=x20b01
CONTROL_CENTER_TOKEN_TTL_MINUTES=30
CONTROL_CENTER_MAX_ATTEMPTS=5
CONTROL_CENTER_LOCK_MINUTES=15
```

Если быстро возможно, вместо plain password сделать hash:

```env
CONTROL_CENTER_PASSWORD_HASH=
```

Важно:

* `CONTROL_CENTER_PASSWORD` или hash должны быть только на backend;
* frontend не должен знать правильный пароль;
* frontend только отправляет введённый пароль на endpoint unlock;
* не логировать пароль;
* не отдавать пароль в public system-state;
* не показывать Control Center обычным пользователям.

Backend.

1. Создать router:
   `src/backend/api/control_center.py`

Подключить его в `src/backend/app.py`.

Endpoint'ы:

`POST /api/control-center/unlock`

Body:

```json
{
  "password": "x20b01"
}
```

Поведение:

* если `CONTROL_CENTER_ENABLED=false`, вернуть 404 или 403;
* проверить пароль;
* проверить rate limit попыток;
* если пароль верный — выдать временный control-token;
* если пароль неверный — вернуть понятную ошибку без раскрытия деталей.

Response:

```json
{
  "control_token": "string",
  "expires_at": "datetime",
  "ttl_minutes": 30
}
```

`POST /api/control-center/lock`

* отзывает текущий token.

`GET /api/control-center/status`

* требует control-token;
* возвращает live status проекта:

  * database connected;
  * total users;
  * active users;
  * blocked users;
  * notifications today;
  * publications count;

  * scenarios count;
  * authorities count;
  * regions count;
  * maintenance mode;
  * readonly mode;
  * banner enabled;
  * AI status;
  * push/native notification status;
  * scheduler status, если возможно;

5. Техническое обслуживание.

Endpoint:
`PUT /api/control-center/maintenance`

Header:
`X-Control-Center-Token`

Body:

```json
{
  "enabled": true,
  "level": "warning|readonly|full_lock",
  "title": "Белпомощник временно на техническом обслуживании",
  "message": "Мы обновляем справочные данные и скоро вернёмся.",
  "until": null,
  "allow_admin_access": true
}
```

Режимы:

* `warning` — сверху показывается предупреждение, функции работают;
* `readonly` — пользователи могут смотреть, но не могут создавать/редактировать;
* `full_lock` — обычные пользователи видят экран технического обслуживания и не могут пользоваться приложением.

Frontend:

* создать компонент `MaintenanceScreen.tsx`;
* при `full_lock` показывать экран техобслуживания;
* при `warning` показывать баннер;
* при `readonly` показывать предупреждение и блокировать формы.

Backend:

* добавить helper для write endpoint'ов:
  `ensure_not_readonly_mode()`;
* при readonly/full_lock блокировать основные пользовательские write actions:

  * создание ситуаций;
  * редактирование ситуаций;
  * документы;
  * сканы;
  * ЖКХ и налоги;
  * профиль;
  * предложения публикаций;
  * AI write-actions, если есть.
