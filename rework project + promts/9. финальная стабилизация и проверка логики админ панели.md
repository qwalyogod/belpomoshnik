Ты работаешь с моим дипломным проектом «Белпомощник».

Задача: после доработки админ-панели провести финальную стабилизацию и проверку логики. Нужно убедиться, что админ действительно контролирует всё, что отображается на сайте, а роли и ограничения работают правильно.

Перед началом прочитай:

* `docs/PROJECT_STATUS.md`
* `docs/API_CONTRACTS.md`
* `docs/SECURITY_POLICY.md`
* `docs/DEMO_SCRIPT.md`
* `src/backend/models.py`
* `src/backend/api/admin.py`
* все admin components во frontend
* `reactvitemaket/src/app/services/api.ts`
* `reactvitemaket/src/app/data/types.ts`

Проверить принцип:
Если сущность видна пользователю на сайте, она должна быть видна администратору в админке.

Проверить разделы:

1. Публикации

* все новости видны;
* все правовые обновления видны;
* черновики видны;
* опубликованные видны;
* архив виден;
* можно изменить статус;
* можно открыть публичную страницу.

2. Проблемы

* все проблемы из базы видны в админке;
* опубликованные и неопубликованные различаются статусом;
* можно создать проблему;
* можно редактировать;
* можно скрыть;
* можно удалить/архивировать;
* можно открыть публичную карточку;
* связанные сценарии отображаются.

3. Сценарии

* все сценарии из базы видны;
* видно количество этапов и шагов;
* можно открыть редактор сценария;
* можно изменить статус;
* можно привязать проблему;
* можно привязать учреждения;
* можно открыть публичную страницу;
* есть проверка целостности сценария.

4. Учреждения

* все учреждения видны;
* поиск работает;
* фильтры по региону/городу работают;
* можно редактировать;
* можно связать со сценариями/проблемами;
* нет битых ссылок на публичной части.

5. Регионы

* все регионы видны на карте или в списке;
* новый регион не пропадает;
* переименованный регион остаётся на старом месте;
* координаты сохраняются после перезагрузки;
* список районов раскрывается удобно;
* desktop/tablet не используют огромный неудобный modal там, где нужен inspector/drawer;
* mobile остаётся удобным.

6. Пользователи и роли

* admin видит всех пользователей;
* admin может назначить роль;
* admin может заблокировать пользователя;
* admin может разблокировать пользователя;
* admin может удалить/деактивировать пользователя;
* editor не может менять роли;
* citizen не может войти в админку;
* нельзя случайно удалить самого себя;
* заблокированный пользователь не может создавать предложения публикаций;
* заблокированный пользователь не может выполнять чувствительные действия.

7. Модерация и предложения

* новые предложения видны;
* можно принять;
* можно отклонить;
* можно вернуть на доработку;
* можно преобразовать в публикацию;
* можно оставить комментарий;
* предложения заблокированных пользователей не создаются.

8. Журнал действий

* изменение роли логируется;
* бан логируется;
* публикация логируется;
* редактирование региона логируется;
* удаление/архивация логируется;
* в журнале есть actor, action, entity, дата.

9. Безопасность

* все admin endpoints требуют JWT;
* все admin endpoints проверяют роль;
* frontend не является единственной защитой;
* обычный пользователь не может обратиться к admin API напрямую;
* редактор не может получить доступ к системным действиям;
* ошибки не раскрывают stack trace.

10. UX

* нет сырых technical labels;
* все статусы переведены на русский;
* destructive actions требуют подтверждения;
* таблицы имеют поиск/фильтры;
* пустые состояния понятные;
* loading/error states есть;
* desktop удобен;
* tablet удобен;
* mobile не сломан.
