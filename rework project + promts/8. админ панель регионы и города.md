Ты работаешь с моим дипломным проектом «Белпомощник».

Задача: полностью переделать раздел админки “Регионы и города”, сохранив красивую карту Беларуси, но сделать её реально удобной и стабильной на desktop, tablet и mobile.

Контекст:
Сейчас раздел выглядит визуально красиво, особенно на мобильных экранах, но на desktop/tablet неудобен:

* карта маленькая и стоит по центру огромной пустой области;
* редактирование открывается большим модальным окном поверх затемнения;
* на desktop/tablet это выглядит как мобильная логика, растянутая на большой экран;
* если добавить новый регион или переименовать существующий, он может пропасть визуально;
* нет понятного списка всех регионов рядом;
* нет управления позицией карточек на карте;
* нет нормального способа раскрыть список районов без модального окна;
* после переименования регион должен оставаться на том же месте;
* новый регион должен появляться в понятном месте, например в правом верхнем углу или в отдельной зоне “Новые регионы без позиции”.

Перед началом прочитай:

* все компоненты, где реализован раздел “Регионы и города”

* миграции регионов, если они есть
* `docs/DESIGN_SYSTEM.md`

Главная цель:
Сделать раздел “Регионы и города” как полноценный редактор географического справочника, а не просто картинку с карточками.

Нужно реализовать новую UX-логику.

1. Desktop layout

На desktop сделать split layout:

Слева:

* большая карта Беларуси;
* занимает основную часть экрана;
* не маленькая карточка по центру;
* масштабируется по ширине контейнера;
* сохраняет пропорции;
* карточки регионов поверх карты;
* карточки можно выбирать;
* выбранный регион подсвечивается.

Справа:

* боковая панель-инспектор выбранного региона;
* если регион не выбран — список всех регионов;
* если регион выбран — редактирование региона и список районов.

Пример логики:

```text
[ Sidebar админки ] [ Карта Беларуси 60-70% ] [ Инспектор 30-40% ]
```

В инспекторе:

* название региона;
* главный город/центр;
* количество районов;
* статус;
* координаты карточки на карте;
* список районов;
* кнопка “Добавить район”;
* кнопка “Сохранить”;
* кнопка “Удалить регион”;
* кнопка “Сбросить позицию”.

Не использовать большой модальный overlay для обычного редактирования на desktop.
Модалки использовать только для подтверждения удаления или опасных действий.

2. Tablet layout

На tablet:

* карта сверху или слева;
* список/инспектор открывается как right drawer;
* если экран достаточно широкий, использовать split layout;
* если узкий, использовать drawer.

Не должно быть огромной серой затемнённой области как на скриншоте.
Редактирование должно ощущаться как рабочая панель, а не как popup поверх пустоты.

3. Mobile layout

На mobile можно оставить текущую хорошую логику:

* карта сверху;
* выбранный регион открывается bottom sheet;
* список районов в bottom sheet;
* кнопки крупные;
* без горизонтального скролла.

Но нужно проверить:

* карточки не перекрывают друг друга слишком сильно;
* можно выбрать любой регион;
* можно закрыть редактирование;
* можно сохранить;
* можно добавить район.

4. Стабильные позиции карточек

Сейчас после добавления/переименования регион может пропасть визуально. Нужно исправить.

Добавить в модель региона поля:

* `map_label_x`
* `map_label_y`
* `map_label_anchor`
* `map_label_order`
* возможно `map_label_visible`

Где:

* `map_label_x` и `map_label_y` — проценты от 0 до 100 относительно контейнера карты;
* не пиксели, чтобы карта адаптировалась;
* при переименовании региона позиция не меняется;
* при создании нового региона позиция ставится по умолчанию:

  * например `x=85`, `y=15`;
  * или в зону “Новые”;
  * или автоматически рядом со списком;
* если координат нет, frontend должен показывать карточку в fallback-зоне, а не скрывать её.

Backend:

* добавить поля в модель/migration;
* endpoints должны возвращать эти поля;
* update endpoint должен сохранять эти поля.

Frontend:

* карточка региона использует `left: ${mapLabelX}%`, `top: ${mapLabelY}%`;
* если координаты отсутствуют — использовать fallback coordinates;
* не рассчитывать позицию по названию региона;
* не привязывать позицию к старому имени.

5. Drag & drop карточек

Добавить возможность двигать карточки регионов на карте.

Поведение:

* включить режим “Редактировать расположение”;
* в этом режиме карточки можно перетаскивать мышью/тачем;
* после перетаскивания координаты сохраняются в процентах;
* кнопки:

  * “Сохранить расположение”;
  * “Отменить”;
  * “Сбросить расположение”.
* вне режима редактирования drag отключён, чтобы случайно не двигать карточки.

Важно:

* drag должен работать на desktop мышью;
* на tablet touch;
* на mobile можно отключить drag или сделать через отдельный режим, чтобы не мешать скроллу.

Если сложно подключать библиотеку — реализуй через pointer events:

* `pointerdown`
* `pointermove`
* `pointerup`
* координаты считать относительно bounding rect карты.

6. Список регионов

Добавить полноценный список регионов.

На desktop:

* список находится в правой панели, если ничего не выбрано;
* либо отдельная вкладка “Список”.
* каждый элемент:

  * название;
  * центр;
  * количество районов;
  * статус;
  * кнопка редактирования;
  * кнопка показать на карте.

На tablet:

* список открывается справа drawer.

На mobile:

* список открывается bottom sheet или отдельным экраном.

Обязательно:

* если регион создан, он сразу появляется в списке;
* если карточка на карте не видна из-за координат, регион всё равно виден в списке;
* из списка можно выбрать регион и центрировать/подсветить его на карте.

7. Редактирование региона

Редактирование должно быть логичным.

Поля:

* название региона;
* отображаемое название на карте;
* центр/главный город;
* статус активности;
* карта: x/y;
* районы.

Район:

* название района;
* центр района;
* статус активности;
* порядок сортировки.

Действия:

* добавить район;
* удалить район;
* переименовать район;
* изменить центр;
* изменить порядок районов;
* сохранить;
* отменить.

Важно:

* при переименовании региона карточка остаётся на том же месте;
* при сохранении изменения сразу отображаются на карте и в списке;
* если сохранение не удалось — показать ошибку и вернуть предыдущее состояние;
* не удалять регион физически, если к нему привязаны пользователи/учреждения/контент, лучше предложить архивировать.

8. Backend validation

Добавить проверки:

* название региона не пустое;
* название региона уникально;
* координаты x/y от 0 до 100;
* район не пустой;
* нельзя удалить регион, если к нему привязаны учреждения/пользователи/сценарии, без специального подтверждения;
* нельзя создать два одинаковых района внутри одного региона.

9. Массовое восстановление стандартных регионов

Добавить кнопку:

* “Восстановить стандартные регионы Беларуси”

Поведение:

* не удалять пользовательские данные без подтверждения;
* восстановить/создать стандартные области:

  * Брестская
  * Витебская
  * Гомельская
  * Гродненская
  * Минская
  * Могилёвская
  * Минск
* проставить стандартные координаты карточек;
* восстановить стандартные списки районов;
* если регион уже есть — обновить только missing districts, если пользователь подтвердил.

Кнопка должна иметь confirmation dialog.

10. Визуальные требования

Карта:

* должна быть крупнее на desktop;
* не должна теряться в огромной белой карточке;
* использовать доступную область;
* подписи должны быть читаемыми;
* карточки регионов не должны улетать за карту;
* выбранная карточка подсвечивается;
* новые/непозиционированные карточки имеют отдельный стиль или fallback-зону.

Инспектор:

* чистый дизайн;
* без огромного overlay на desktop;
* sticky header;
* sticky save bar снизу;
* список районов со скроллом;
* кнопка удаления визуально отделена.
