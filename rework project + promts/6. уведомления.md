Ты работаешь с моим дипломным проектом «Белпомощник».

Задача: доработать систему уведомлений с учётом того, что проект работает как сайт и как мобильное/tablet-приложение через WebView/Capacitor. Всплывающие уведомления не должны рисоваться вручную на frontend. Нужно использовать готовые системные механизмы браузера или операционной системы. Если пользователь не дал разрешение на уведомления, ничего не должно всплывать — уведомления должны тихо сохраняться внутри приложения во вкладке “Уведомления”.

Главная логика:

1. Любое уведомление сначала создаётся на backend и сохраняется в аккаунте пользователя.
2. Вкладка “Уведомления” внутри приложения работает всегда.
3. Если пользователь не разрешил уведомления:

   * не показывать всплывающие уведомления;
   * не пытаться обходить запрет;
   * просто сохранять уведомления внутри приложения.
4. Если пользователь разрешил уведомления:

   * на desktop web использовать браузерные уведомления/Web Push;
   * на Android/iOS/tablet app использовать нативные уведомления через Capacitor;
   * не рисовать кастомные toast/popup на frontend как замену системным уведомлениям.
5. WebView не должен считаться надёжным механизмом доставки, потому что когда приложение закрыто, WebView не работает.
6. Для событий по срокам можно использовать локальные уведомления на устройстве.
7. Для внезапных серверных событий нужен push-токен и backend-доставка через FCM/APNs, но если credentials пока не настроены — сделать архитектуру, endpoints и документацию, а не имитировать работу.

Нужно реализовать три уровня уведомлений.

Уровень 1. In-app notification center.

Это основа. Работает всегда.

Проверь/доработай модель `UserNotification`:

* `id`
* `user_id`
* `title`
* `description`
* `notification_type`
* `source`
* `route`
* `due_date`
* `is_read`
* `created_at`
* `dedupe_key`

Если `dedupe_key` нет — добавь migration.

Правила:

* уведомления не должны дублироваться бесконечно;
* все уведомления видны во вкладке “Уведомления”;
* mark read / mark all read / delete должны работать;
* если пользователь запретил push, in-app уведомления всё равно создаются.

Уровень 2. Local Notifications для mobile/tablet app.

Добавь поддержку `@capacitor/local-notifications`.

Использовать только если приложение запущено как Capacitor/native app.

Создай:

* `reactvitemaket/src/app/services/nativeNotifications.ts`

Функции:

* `isNativeApp()`
* `checkNativeNotificationPermission()`
* `requestNativeNotificationPermission()`
* `scheduleLocalNotification(reminder)`
* `scheduleUpcomingLocalNotifications(reminders)`
* `cancelLocalNotification(id)`
* `syncNativeNotificationSchedule()`

Логика:

* после входа пользователя приложение загружает будущие уведомления/напоминания;
* если пользователь разрешил уведомления на устройстве, приложение планирует локальные уведомления;
* если пользователь запретил — ничего не планирует;
* локальные уведомления использовать для:

  * срока документа;
  * срока задачи;
  * ЖКХ;
  * налогов;
  * запланированных пользовательских напоминаний.
* локальные уведомления не использовать для внезапных серверных событий, потому что приложение может быть закрыто и не узнать о них вовремя.

Важно:

* не показывать уведомление через кастомный frontend popup;
* использовать именно системный механизм Capacitor Local Notifications;
* не включать разрешения автоматически — только по нажатию пользователя.

Уровень 3. Native Push token для будущей production-доставки.

Добавь архитектурную поддержку `@capacitor/push-notifications`.

Если безопасно для сборки, добавь dependency:

* `@capacitor/push-notifications`

Создай:

* `reactvitemaket/src/app/services/nativePush.ts`

Функции:

* `isNativeApp()`
* `checkPushPermission()`
* `requestPushPermission()`
* `registerNativePush()`
* `sendPushTokenToBackend(token, platform)`
* `unregisterNativePushToken()`

Backend:
создай/доработай endpoint'ы:

* `POST /api/user/push/native-token`
* `DELETE /api/user/push/native-token`
* `GET /api/user/push/status`
* `POST /api/user/push/test`

Модель:
`UserPushToken`

* `id`
* `user_id`
* `platform`: `ios` / `android`
* `token_hash`
* `token_encrypted` или `token`
* `device_label`
* `is_active`
* `last_seen_at`
* `created_at`
* `updated_at`

Важно:

* token принадлежит только текущему пользователю;
* другой пользователь не может читать/менять чужой token;
* полный token не возвращать на frontend;
* если credentials FCM/APNs не настроены, backend должен возвращать `native_push_ready=false`, а не падать.

Доставка:

* если FCM/APNs credentials не настроены, не имитировать отправку;
* сделать заглушку service layer:

  * `src/backend/notifications/native_push.py`
  * функция `send_native_push(...)`
  * если credentials нет — статус `skipped`, причина `credentials_not_configured`;
* в документации написать, что production-доставка требует:

  * Android: Firebase Cloud Messaging и `google-services.json`;
  * iOS: Apple Developer Account, APNs capability/key/certificate.

Правила:

Push payload.

Payload должен быть безопасным:

```json
{
  "title": "Белпомощник",
  "body": "У вас есть новое уведомление",
  "notificationId": "123",
  "type": "document_expiry",
  "route": "/notifications"
}
```

Не отправлять:

* номера документов;
* адреса;
* сканы;
* паспортные данные;
* точные персональные подробности.

Frontend sync.

При запуске приложения:

1. определить платформу:

   * web desktop;
   * web mobile;
   * Capacitor Android;
   * Capacitor iOS;
   * tablet.
2. загрузить `/api/user/push/status`;
3. загрузить preferences;
4. если native app и разрешения есть:

   * зарегистрировать push token;
   * отправить token на backend;
   * синхронизировать локальные уведомления по будущим срокам.
5. если разрешения нет:

   * не просить автоматически;
   * показать кнопку включения на странице уведомлений/настройках.
6. при logout:

   * не обязательно удалять token сразу, но лучше деактивировать текущий device token;
   * отменить локальные уведомления, если они относятся к этому пользователю.

Тестовое уведомление.

`POST /api/user/push/test` должно:

* создать in-app уведомление “Тестовое уведомление”;
* если разрешён native push/browser push — попытаться доставить системно;
* если не разрешён — вернуть:

  * `in_app_created: true`
  * `system_delivered: false`
  * `reason: "permission_not_granted"`

Frontend должен показать понятное сообщение:

* “Тестовое уведомление сохранено во вкладке уведомлений.”
* “Системное уведомление не отправлено, потому что разрешение не выдано.”
