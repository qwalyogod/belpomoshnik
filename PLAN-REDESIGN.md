# PLAN-REDESIGN.md — единое ТЗ редизайна «Бел Помощник»

Файл — единственный источник истины для агента-разработчика (Codex) на время редизайна. Читать целиком перед каждой итерацией. Источники макетов и текстового ТЗ — в папке `Бел Помощник redesign сырые макеты/` в корне репозитория (20 PNG/SVG + 2 docx).

---

## 1. Контекст и цели

### Стек
- Flet 0.85.0 (Python 3.14) — единственный UI-фреймворк. React/HTML/CSS НЕ использовать.
- Бэкенд FastAPI + SQLAlchemy + SQLite (`src/backend/`), API на `http://127.0.0.1:8060`.
- Монолитный `src/main.py` (~3000 строк) — навигация, state, callback-и админки.
- Страницы в `src/pages/` (18 файлов), общие компоненты в `src/components/`, тема в `src/theme/app_theme.py`.
- Запуск web: `.venv/bin/flet run -w src/main.py --host 127.0.0.1 -p 8550`.
- Бэкенд: `PYTHONPATH=src .venv/bin/uvicorn backend.app:app --host 127.0.0.1 --port 8060`.
- Синтакс-проверка: `.venv/bin/python -m compileall src`.

### Что есть сейчас
- Цвета жёстко в `src/theme/app_theme.py` → словарь `APP_COLORS`, **только светлая тема** (`page.theme_mode = ft.ThemeMode.LIGHT`).
- Свежий код (близко к финалу визуально): админка (`admin_page.py`, A1–A12 завершены 2026-05-24), `home_page.py`, `problem_detail_page.py`, `situations_page.py`, `scenario_detail_page.py`, `legal_updates_page.py`, `documents_page.py`.
- Старые прототипы (упрощённый UI, требуют полной переделки): `learning_progress_page.py`, `problems_page.py`, `login_page.py`, `register_page.py`, `onboarding_page.py`, `about_page.py`, `notifications_page.py`.

### Чего достигаем
- Единая дизайн-система: цвета, типографика, скругления, отступы, тени — токенами.
- **Telegram-inspired civic assistant**: сайдбар-каналы на desktop, нижний таб-бар на mobile, контент в виде структурированных карточек/статей.
- Поддержка **обеих тем**: light (фоны почти белые с лёгкой голубизной) и dark (синий-графитовый, **не чёрный**), переключатель сохраняется в состоянии.
- Адаптивность: desktop ≥1180px и mobile <1180px — разные layout'ы, не «уменьшенный desktop».
- Сдержанные анимации в духе iOS/macOS (`EASE_OUT`, 200–300ms), без спама.
- Плейсхолдеры под фото — обведённые прямоугольники с подписью «здесь будет фото: …». Никаких сгенерированных картинок.

### Что НЕ делаем
- Чатовые пузыри, ощущение «сообщений от кого-то», «отправители».
- Серая казённая госуслуговая палитра.
- Mobile как сжатый desktop — у mobile своя сетка и навигация.
- Острые углы (radius < 10px) — везде округло.
- Новые маршруты/функции/таблицы БД — только визуальный редизайн.
- Переписывание бизнес-логики админки A1–A12 (`admin_page.py`, callback-и в `main.py`) — только визуальный слой.
- Перенос на PostgreSQL / JWT / RBAC / любые этапы G–K из `claude/PLAN.md` — вне scope.

## 3. Архитектура темы и переключателя

### 3.1. `src/theme/app_theme.py` — переписать
- Экспортировать `LIGHT`, `DARK`, `BADGE_LIGHT`, `BADGE_DARK`, `RADIUS`, `SPACING`, `SHADOWS`.
- Функция `get_theme(mode: str) -> dict` возвращает соответствующий словарь (`"light"` или `"dark"`).
- Функция `get_badge_palette(mode: str) -> dict`.
- **Обратная совместимость:** оставить экспорт `APP_COLORS` — словарь-прокси, который при чтении ключа возвращает значение из активной темы. Реализация: класс с `__getitem__`, читающий текущий режим из `page.session.get("theme_mode")` или из модульного state. Это критично — старые ключи (`primary`, `background`, `surface`, `surface_alt`, `text`, `muted`, `primary_light`, `primary_dark`, `blue`, `secondary`, `warning`, `danger`) сейчас используются десятками раз в `src/pages/*.py` и `src/main.py`. Замапить старые ключи на новые: `primary` → `blue`, `background` → `screen`, `surface_alt` → `surface2`, `primary_dark` → `blue_text`, `primary_light` → `active`, `secondary` → `green`, `warning` → `orange`, `danger` → `red`. Маппинг — внутри прокси, чтобы старые страницы продолжали работать пока их не переделают.
- Функция `card_shadow(mode)` — возвращает `ft.BoxShadow` по теме.

### 3.2. State и сохранение
- В `src/main.py`: `default_state["theme_mode"] = "light"`, читается из `data/app_state.json`.
- При запуске: `page.session.set("theme_mode", app_state["theme_mode"])`, `page.theme_mode = ft.ThemeMode.LIGHT|DARK`.
- Колбэк `toggle_theme()` меняет значение, сохраняет state, вызывает `route_change()` для ре-рендера.

### 3.3. Переключатель
- В сайдбаре (desktop) — нижняя «hint card» с иконкой солнца/луны.
- В профиле — отдельная строка настроек с `ft.Switch`.

### 3.4. Использование в компонентах
Компоненты `src/components/*.py` читают цвета через `APP_COLORS["key"]` (прокси сам подставит активную тему) либо принимают `theme: dict` опционально. Без жёстко зашитых hex в страницах.

## 6. Правила работы Codex

### 6.1. Алгоритм каждой итерации

1. Прочитать `PLAN-REDESIGN.md` целиком (этот файл) — особенно секции 2, 4, 6.
2. Определить **текущий шаг** = первый без чек-листа `✅` в секции «Прогресс редизайна» этого файла (см. ниже).
3. Открыть PNG/SVG из соответствующей папки макетов через Read tool — внимательно описать что видит (desktop+mobile, light+dark).
4. Прочитать целевой файл(ы) полностью.
5. Если шаг = 0 (дизайн-система) — переписать `src/theme/app_theme.py` и компоненты в `src/components/`. Если шаг 1–20 — править страницу.
6. Соблюдать:
   - только токены из секции 2 (никаких произвольных hex);
   - обе темы (light + dark) корректно отображаются;
   - desktop layout (sidebar + content) и mobile layout (topbar + content + bottom_nav) — оба работают;
   - вместо фото — `PhotoPlaceholder` с описательной подписью;
   - анимации: `ft.animation.Animation(220, ft.AnimationCurve.EASE_OUT)` на cards, `ft.AnimatedSwitcher(transition=ft.AnimatedSwitcherTransition.FADE, duration=250)` на смену секций;
   - русский UI, единый словарь терминов.
7. Запустить `.venv/bin/python -m compileall src` — без ошибок.
8. Запустить `.venv/bin/flet run -w src/main.py --host 127.0.0.1 -p 8550`, открыть страницу, проверить визуально обе темы и оба устройства (resize окна).
9. Сделать 4 скриншота: `docs/redesign-progress/<номер>-desktop-light.png`, `<номер>-desktop-dark.png`, `<номер>-mobile-light.png`, `<номер>-mobile-dark.png`. Создать папку `docs/redesign-progress/` если её нет.
10. Обновить чек-лист соответствующего шага в этом файле (раздел «Прогресс редизайна» — отметить ✅ напротив текущего шага).
11. Обновить `claude/PLAN.md` (добавить отметку в раздел «Redesign progress») и `docs/CHANGELOG.md` (короткая запись «Redesign: <экран> готов»).
12. Отчитаться (формат — секция 7.4) и **остановиться**, ожидая ревью.
